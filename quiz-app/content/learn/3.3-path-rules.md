## What are path-specific rules?

In the CLAUDE.md hierarchy (lesson 3.1), all files load at session start and apply everywhere. But not all conventions apply everywhere — your testing rules only matter when Claude is touching test files, and your Terraform conventions only matter when working in the infrastructure directory.

Path-specific rules solve this. A rule file with a `paths:` frontmatter declaration loads *on demand*, only when Claude reads a file matching those patterns. This means test rules load when Claude opens `auth.test.ts` but stay out of context when Claude is working on API code.

## How it works

Path-specific rules live in `.claude/rules/`. Files *without* `paths:` frontmatter load at session start (same as CLAUDE.md). Files *with* `paths:` frontmatter load conditionally:

```
.claude/
├── CLAUDE.md              ← universal rules, always loaded
└── rules/
    ├── testing.md         ← loads when Claude reads test files
    ├── api-conventions.md ← loads when Claude reads API files
    ├── security.md        ← loads when Claude reads any source file
    └── deployment.md      ← loads when Claude reads terraform/
```

A rule file with paths frontmatter looks like this:

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules

All API endpoints must include input validation.
Use the standard error response format: { error: string, code: string, details?: object }
Include OpenAPI documentation comments on every handler.
```

Another example — test conventions:

```markdown
---
paths:
  - "**/*.test.tsx"
  - "**/*.spec.ts"
  - "tests/**/*"
---

# Testing Conventions

Use describe/it block structure.
Mock external services with jest.mock().
Every test file must have at least one assertion.
```

## Why path-specific rules beat directory-level CLAUDE.md for scattered files

Test files in most projects are scattered throughout the codebase — `Button.test.tsx` lives next to `Button.tsx`, `auth.test.ts` lives next to `auth.ts`. A directory-level `CLAUDE.md` in `tests/` won't cover any of these co-located files.

A `.claude/rules/testing.md` with `paths: ["**/*.test.tsx", "**/*.spec.ts"]` covers *all* test files regardless of where they live.

## Glob patterns for paths frontmatter

| Pattern | What it covers |
|---|---|
| `terraform/**/*` | All files under terraform/ |
| `**/*.test.tsx` | All test files anywhere in the repo |
| `src/components/*.tsx` | Components in one specific directory |
| `*.md` | Markdown files at the project root |
| `src/**/*.{ts,tsx}` | All TypeScript files under src/ |

## Key things to know

- Rules files **with** `paths:` frontmatter load conditionally when Claude reads matching files
- Rules files **without** `paths:` frontmatter load at session start, every time
- Path-specific rules are ideal for cross-directory conventions (test files, config files) that can't be covered by a single directory-level CLAUDE.md
- Path-specific rules are **lost after `/compact`** — they re-load only when Claude reads a matching file again
- Multiple `paths:` values are OR'd — a file matching any pattern triggers the rule

## Exam tips

- A question about "test conventions applying across the whole codebase" → path-specific rules in `.claude/rules/`, not a directory-level CLAUDE.md
- "Rules with `paths:` frontmatter load at launch" — **false**. They load on demand when matching files are read
- After `/compact`, path-scoped rules are lost — this is a common question about what survives compaction
- The exam may distinguish between `.claude/rules/` files with and without `paths:` — know the difference clearly
