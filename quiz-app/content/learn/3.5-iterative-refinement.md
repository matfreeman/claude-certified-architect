## What is iterative refinement?

Getting Claude to produce high-quality output rarely happens in a single prompt. Complex tasks — especially those with precise formatting requirements, edge cases, or ambiguous requirements — benefit from an iterative approach: start, inspect, correct, repeat.

The techniques in this lesson aren't about Claude's limitations. They're about *communication*. The same principles that help you work effectively with a human engineer apply here: concrete examples beat vague descriptions, visible test failures give specific targets, and asking questions upfront prevents rework.

## Technique 1: Concrete input/output examples

When prose instructions produce inconsistent results, replace them with examples:

```
BAD:  "Transform dates to a consistent format"

GOOD: Show concrete examples:
  Input:  "January 15th, 2025" → Output: "2025-01-15"
  Input:  "15/01/25"           → Output: "2025-01-15"
  Input:  "Jan 15 2025"        → Output: "2025-01-15"
```

Claude reads the examples and infers the rule. This is more reliable than a description because edge cases in the description are implicit, but edge cases in examples are explicit.

## Technique 2: Test-driven iteration

Write the tests first, then ask Claude to implement against them:

```
1. Write test suite covering expected behaviour, edge cases, performance
2. Share the implementation prompt + tests
3. Claude implements
4. Run tests — inspect which pass and which fail
5. Share specific failures back to Claude
6. Repeat until all tests pass
```

The key insight: test failures give Claude *specific, verifiable targets*. "It doesn't work" is hard to act on. "This assertion fails: expected 'YYYY-MM-DD', got 'January 15th, 2025'" is immediately actionable.

A specific failing case is always better than a general complaint:

```
The migration script handles null values incorrectly.

Specific failing case:
  Input:    {"user_id": 123, "preferences": null, "created_at": "2025-01-01"}
  Expected: {"user_id": 123, "preferences": {}, "created_at": 1735689600}
  Actual:   TypeError: Cannot read property 'theme' of null
```

## Technique 3: The interview pattern

For tasks in unfamiliar domains, ask Claude to surface its questions *before* implementing:

```
Instead of: "Implement a Redis cache for our user sessions"

Try: "Before implementing a Redis cache for user sessions, 
      ask me the questions you'd need answered to make the right design decisions"
```

Claude will identify the decisions that actually matter:
- What's the cache invalidation strategy when users update their profile?
- Should sessions survive Redis restarts?
- What's the expected session size?
- Is distributed locking needed for concurrent updates?

Answering these before implementation prevents the most expensive rework — the kind that changes the architecture.

## Single message vs sequential iteration

| Issue type | Approach |
|---|---|
| Issues that interact (fixes affect each other) | One detailed message with all issues |
| Independent issues | Sequential — fix one, verify, then the next |

Example of interacting issues: "Fix the null pointer exception AND refactor the validation logic." These interact because the null check is probably *inside* the validation logic. A sequential fix could break the first fix when you make the second change.

## Key things to know

- Examples outperform prose descriptions for format-sensitive tasks
- Test failures give Claude specific, actionable targets — always prefer them over vague corrections
- The interview pattern prevents expensive rework by surfacing design questions before implementation
- Fix interacting issues in one message; fix independent issues sequentially
- Inspecting intermediates (partial outputs, validation artifacts) at each iteration step is part of the workflow, not optional

## Exam tips

- A question about inconsistent date formatting → the fix is concrete input/output examples, not more descriptive prose
- "Claude produced the wrong architecture" → the interview pattern (ask questions first) prevents this, not more detailed requirements prose
- Interacting vs. independent issues: the exam will test whether you know when to batch fixes vs. sequence them
- Test-driven iteration is explicitly a recommended Claude Code workflow — know the sequence
