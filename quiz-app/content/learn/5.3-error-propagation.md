## What is error propagation in multi-agent systems?

In a multi-agent system, subagents run tasks in parallel and return results to a coordinator. When one subagent fails, the coordinator has to decide: retry, use partial results, use an alternative, or abort the whole workflow?

That decision is only possible if the subagent communicates *why* it failed and *what options exist*. A bare `{"error": "search unavailable"}` gives the coordinator nothing to work with. A structured error with type, partial results, and recovery suggestions gives the coordinator everything it needs.

The core principle: **errors are data**. Treat them with the same care as successful results.

## How it works

```
┌──────────────┐   success   ┌─────────────┐
│  Subagent    │ ──────────→ │ Coordinator │
│  (searcher)  │             │             │
│              │ structured  │  decides:   │
│              │ error ────→ │  - retry?   │
└──────────────┘             │  - partial? │
                             │  - abort?   │
                             └─────────────┘
```

Compare a bare error vs. a structured error:

```python
# WRONG — coordinator can't make recovery decisions
return {"error": "search unavailable"}

# CORRECT — structured context with recovery options
return {
    "success": False,
    "error": {
        "type": "transient",
        "message": "Web search timeout after 10s",
        "attempted_query": "AI impact on music industry 2024 2025",
        "partial_results": [
            {"title": "AI in Music Production", "url": "...", "snippet": "..."}
        ],
        "suggested_alternatives": [
            "Retry with simpler query: 'AI music industry'",
            "Use cached research papers via document_analysis subagent",
            "Proceed with partial results and annotate gaps in synthesis"
        ],
        "retry_recommended": True
    }
}
```

## Coordinator recovery logic

With a structured error, the coordinator can make intelligent recovery choices:

```python
async def coordinator_with_recovery(topic: str):
    result = await invoke_web_searcher(topic)
    
    if not result["success"]:
        error = result["error"]
        
        if error["type"] == "transient" and error["retry_recommended"]:
            result = await invoke_web_searcher(simplify_query(topic))
        
        if not result["success"]:
            # Proceed with partial results, annotate the gap
            partial = error.get("partial_results", [])
            return await invoke_synthesizer(
                data=partial,
                gap_annotation="[GAP: web search unavailable for this subtopic]"
            )
```

## Anti-patterns to avoid

| Anti-pattern | What happens |
|---|---|
| **Silent suppression** | `{"results": []}` when search failed — coordinator thinks no data exists, proceeds with an incomplete and misleading output |
| **Workflow abortion** | One subagent failure aborts the entire research task — all parallel work is wasted |
| **Generic messages** | `"search failed"` hides the error type, partial results, and recovery options |

## Coverage annotations in synthesis output

When a workflow completes with gaps, the output should say so explicitly:

```python
synthesis_output = {
    "report": "...",
    "coverage_summary": {
        "visual_arts": "Well-supported — 12 sources found",
        "music_industry": "Partial — web search unavailable, 3 cached papers used",
        "film_production": "GAP — no sources available for this subtopic"
    },
    "confidence": "medium"   # Reduced due to coverage gaps
}
```

A downstream consumer (or a human reviewer) can then make informed decisions about whether the gaps are acceptable or require follow-up.

## Key things to know

- Structured errors enable recovery; bare error strings do not
- Partial results should be included in error responses — they have value even if the full query failed
- One subagent failure should never abort the entire workflow unless the failed result is truly blocking
- Coverage annotations in the output communicate gaps to downstream consumers and human reviewers
- Error type classification (transient vs. business vs. permanent) determines which recovery strategy applies

## Exam tips

- "Coordinator proceeds as if no data exists after a subagent failure" → silent suppression (returning empty results instead of structured error)
- Retry with a simplified query → appropriate for transient errors; not appropriate for "data not found"
- The correct response to a partial failure is partial results + gap annotation, not workflow abort
- Coverage annotations communicate reliability; removing them hides known limitations from users
