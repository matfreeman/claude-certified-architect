## What is information provenance?

When a multi-agent research system gathers findings from multiple sources and synthesises them into a report, each claim in that report should be traceable back to its source. This traceability is *provenance*.

Without it, synthesised reports have a critical weakness: you can't tell which claims are well-supported, which come from a single low-quality source, and which represent conflicting evidence from different time periods. The report looks authoritative but can't be audited or corrected.

The techniques in this lesson ensure that provenance is preserved from the moment a subagent finds a piece of information through to the final synthesised output.

## Why provenance gets lost

Provenance is lost during summarisation. A subagent reads a study and returns: *"AI reduces music production time by 40%."* Gone are: the source URL, the publication date, the sample size, the methodology. The synthesiser receives a claim with no context.

The fix: require subagents to return structured claim-source mappings, not prose summaries.

## Structured claim-source mappings

Subagents should always return this structure — never free-text summaries:

```python
finding = {
    "claim": "AI tools reduced music production time by 40% on average",
    "source": {
        "url": "https://musictech.report/ai-2025",
        "title": "AI in Music Production: 2025 Industry Report",
        "author": "Music Technology Research Group",
        "publication_date": "2025-02-15",    # Required for temporal analysis
        "methodology": "Survey of 500 professional music producers"
    },
    "evidence_excerpt": "Respondents reported an average reduction of 40% (±8%) in production time...",
    "confidence": "high"
}
```

The `publication_date` is not optional. Two studies might show different adoption rates not because they contradict each other, but because one is from 2022 and one is from 2025. Without dates, the synthesiser presents a "conflict" where there is only a time difference.

## Handling conflicting statistics

When two sources give different numbers for the same metric, the synthesiser must annotate the conflict — not pick one:

```
WRONG: "AI adoption in creative industries is 34%"
  (arbitrarily picked one value, discarding the other)

CORRECT:
AI adoption in creative industries:
- Smith et al. (2024): 34% — survey of professional musicians [n=500]
- Johnson & Lee (2025): 52% — music platform analytics [n=1200]

These statistics appear to conflict. Possible explanations:
1. Different definitions of "adoption" (tried once vs. regular use)
2. Different populations (professional vs. semi-professional)
3. 12-month gap — rapid adoption growth between studies
These require reconciliation before citing a single figure.
```

## Report structure for transparency

A well-structured synthesis report separates established findings from contested ones:

```markdown
# AI Impact on Creative Industries — Research Synthesis

## Well-Established Findings (multiple consistent sources)
- AI tools are widely adopted for image generation in advertising (8 sources)
- AI music tools require significant human curation (6 sources)

## Findings Under Active Research (fewer or conflicting sources)
- Economic impact on professional musicians: conflicting data (see conflicts section)
- Long-term effects on creative skill development: insufficient research

## Source Conflicts
[Detail each conflict with full source attribution]

## Coverage Gaps
- Film production: only 2 sources, both from 2023 — current data unavailable
- Fine art: web search unavailable — 3 cached papers from 2022 used
```

## Rendering format by content type

Different content types communicate differently when formatted differently:

| Content type | Recommended format |
|---|---|
| Financial data, statistics | Tables — numbers must align for comparison |
| Narrative, news, quotes | Prose — context and flow matter |
| Technical findings, steps | Structured lists with headers |
| Statistical comparisons | Side-by-side tables with source annotations |

Putting financial data in prose makes it hard to compare. Putting narrative in a table loses context. Match the format to how the content is actually used.

## Key things to know

- Provenance is lost when subagents return prose summaries instead of structured claim-source mappings
- Publication dates are required — many apparent conflicts are actually temporal (old study vs. new study)
- Conflicting statistics must be annotated as conflicts, not resolved by arbitrarily picking one value
- Coverage gaps in the report communicate what's missing — removing them is dishonest
- Rendering format should match how the content is actually used (tables for comparison, prose for narrative)

## Exam tips

- "Synthesiser presents two studies as conflicting" → check publication dates first — it may be a temporal difference, not a real conflict
- "Subagent returns a text summary" → loses provenance; the fix is structured claim-source output
- Coverage gap annotations are a sign of *quality*, not weakness — the exam treats their removal as a flaw
- Rendering format questions: financial/statistical data → tables; narrative → prose
