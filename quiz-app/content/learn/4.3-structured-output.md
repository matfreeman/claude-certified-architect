## What is structured output via tool use?

When you need Claude to return data in a specific format — an extracted invoice, a classified issue, a parsed record — you have two options: ask Claude to format its text response as JSON, or use the **tool use mechanism** with a JSON schema.

The tool use approach is significantly more reliable. Asking Claude to "respond as JSON" still produces plain text that can have syntax errors, unescaped characters, or missing brackets. Defining an extraction tool with a JSON schema means the API validates the output structure before it reaches your application — syntax errors are impossible.

## How it works

Define a tool that represents the structure you want to extract. Then force Claude to call it:

```python
import anthropic

client = anthropic.Anthropic()

extraction_tool = {
    "name": "extract_invoice_data",
    "description": "Extracts structured data from invoice documents",
    "input_schema": {
        "type": "object",
        "properties": {
            "invoice_number": {
                "type": "string",
                "description": "Invoice identifier"
            },
            "total_amount": {
                "type": ["number", "null"],   # nullable — may not exist in document
                "description": "Total invoice amount. Null if not found."
            },
            "currency": {
                "type": ["string", "null"],
                "description": "ISO 4217 currency code (USD, EUR, etc.) or null if unclear"
            },
            "payment_terms": {
                "type": "string",
                "enum": ["net-30", "net-60", "net-90", "immediate", "other"],
                "description": "Payment terms classification"
            },
            "payment_terms_detail": {
                "type": ["string", "null"],
                "description": "If payment_terms is 'other', describe the actual terms here"
            }
        },
        "required": ["invoice_number", "payment_terms"]
    }
}

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[extraction_tool],
    tool_choice={"type": "tool", "name": "extract_invoice_data"},  # Force this tool
    messages=[{"role": "user", "content": invoice_text}]
)

# Read from the tool_use block in the response
for block in response.content:
    if block.type == "tool_use":
        extracted_data = block.input
```

## tool_choice controls whether the tool runs

| Value | Behaviour |
|---|---|
| `"auto"` | Claude decides — may return prose instead of calling the tool |
| `"any"` | Must call a tool (any defined tool) |
| `{"type": "tool", "name": "..."}` | Must call this specific tool — guaranteed extraction |

Use `{"type": "tool", "name": "extract_invoice_data"}` whenever you need mandatory extraction. Use `"any"` when you have multiple schemas and want Claude to choose based on document type.

## Nullable fields prevent hallucination

When a field is required but the data isn't in the document, Claude will fabricate something to satisfy the requirement. Making optional fields nullable gives Claude an honest "not found" path:

```json
// BAD — required field forces Claude to make something up
"vendor_address": {"type": "string"}

// GOOD — nullable allows honest null when data is absent
"vendor_address": {"type": ["string", "null"]}
```

## The "other" + detail pattern for extensible enums

Closed enums reject values not in the list. The `"other"` + detail pattern handles anything outside the known set without losing information:

```json
{
  "payment_terms": {
    "type": "string",
    "enum": ["net-30", "net-60", "net-90", "immediate", "other"]
  },
  "payment_terms_detail": {
    "type": ["string", "null"],
    "description": "Required when payment_terms is 'other' — describe the actual terms"
  }
}
```

Similarly, use `"unclear"` in enums to handle ambiguous cases rather than forcing a wrong classification:

```json
"sentiment": {
  "type": "string",
  "enum": ["positive", "negative", "neutral", "unclear"]
}
```

## What schemas guarantee — and what they don't

```
✅ Eliminate JSON syntax errors
✅ Guarantee required fields are present
✅ Guarantee enum values are from the allowed set
❌ Do NOT prevent semantic errors (line items don't sum to stated total)
❌ Do NOT prevent misclassification (correct field, wrong value)
```

Schemas guarantee *structure*. They don't guarantee *accuracy*. Semantic validation (does the math add up?) requires code, not just a schema.

## Extended thinking + tool use constraints

If you're using extended thinking (budget tokens) alongside tool use:
- Only `tool_choice: "auto"` and `tool_choice: "none"` are compatible
- Forced tool use (`"any"` or a specific tool) is incompatible and returns an error
- Pass the `thinking` blocks from assistant messages back unmodified when returning tool results — omitting them breaks reasoning continuity

## Key things to know

- Tool use + JSON schema eliminates syntax errors that "respond as JSON" cannot prevent
- Forcing the tool with `{"type": "tool", "name": "..."}` is the most reliable extraction pattern
- Nullable fields prevent hallucination when optional data is absent from the source document
- The `"other"` + detail pattern handles values outside a closed enum without losing information
- Schemas guarantee structure, not semantic accuracy — you still need validation for business logic

## Exam tips

- "Claude sometimes returns prose instead of JSON" → `tool_choice: "any"` or force a specific tool; `"auto"` allows prose
- Required non-nullable field on sparse documents → expect hallucinated values; fix with nullable
- Extended thinking + forced tool use → incompatible, returns an API error
- "The schema prevents all extraction errors" → false; schemas prevent syntax and structure errors, not semantic errors
