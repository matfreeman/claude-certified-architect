## What this task statement is really about

The official hooks docs cover a wide range of lifecycle events. The exam guide's Task Statement 1.5 is narrower.

For this certification, **Agent SDK hooks** mainly means:

- intercepting an outgoing tool call before it runs
- normalizing or annotating a tool result after it returns
- choosing hooks instead of prompt-only instructions when a rule must hold every time

So the center of gravity here is **`PreToolUse` + `PostToolUse`**, not memorizing every hook event in the full reference.

Think of hooks as middleware around agent actions. They let you keep cross-cutting concerns like policy enforcement and data cleanup out of the prompt and out of individual tool implementations.

## Exam-relevant hook events

| Hook | Why it matters for Task Statement 1.5 |
|---|---|
| `PreToolUse` | Block, allow, or rewrite a tool call before execution |
| `PostToolUse` | Add normalized context after a tool returns |
| `PostToolUseFailure` | Useful operationally, but secondary to the exam objective |
| Other hook events | Real and documented, but not the main focus of this task statement |

## How the core pattern works

```
Claude decides to call a tool
          │
          ▼
   ┌─────────────┐
   │ PreToolUse  │  ← Your hook fires here
   │   hooks     │    Can: allow, deny, modify input
   └──────┬──────┘
          │  (if allowed)
          ▼
   ┌─────────────┐
   │    Tool     │
   │  executes   │
   └──────┬──────┘
          │
          ▼
   ┌─────────────┐
   │ PostToolUse │  ← Your hook fires here
   │   hooks     │    Can: normalize output, audit, trigger side effects
   └──────┬──────┘
          │
          ▼
    Result returned to Claude
```

`PreToolUse` fires before execution. `PostToolUse` fires after execution. That simple before/after split is what the exam keeps returning to:

- **before execution**: enforce a hard rule
- **after execution**: clean up what the model will see next

## In practice

### PreToolUse: blocking policy violations

Use `PreToolUse` when the business rule must be deterministic. A good exam example is blocking refunds above an auto-approval threshold:

```python
async def enforce_refund_policy(input_data, tool_use_id, context):
    """Block refunds exceeding $500 auto-approval threshold."""
    if input_data["hook_event_name"] != "PreToolUse":
        return {}

    if input_data["tool_name"] != "process_refund":
        return {}

    amount = input_data["tool_input"].get("amount", 0)
    if amount > 500:
        return {
            "systemMessage": (
                "Refunds above $500 require human approval. "
                "Use escalate_to_human instead."
            ),
            "hookSpecificOutput": {
                "hookEventName": input_data["hook_event_name"],
                "permissionDecision": "deny",
                "permissionDecisionReason": (
                    f"Refund of ${amount} exceeds the auto-approval threshold."
                )
            }
        }

    return {}
```

When the hook returns `permissionDecision: "deny"`, the tool does not run. Claude gets the reason and can adapt, for example by escalating to a human workflow.

### PostToolUse: normalizing heterogeneous data

Use `PostToolUse` when multiple tools return inconsistent shapes or formats. This maps directly to the exam guide's example of Unix timestamps, ISO 8601 values, and numeric status codes:

```python
async def normalize_tool_outputs(input_data, tool_use_id, context):
    """Normalize date formats and status codes from different MCP tools."""
    if input_data["hook_event_name"] != "PostToolUse":
        return {}

    tool_output = dict(input_data.get("tool_response", {}))

    # Unix timestamp → ISO 8601
    if "timestamp" in tool_output and isinstance(tool_output["timestamp"], int):
        from datetime import datetime, timezone
        tool_output["timestamp"] = datetime.fromtimestamp(
            tool_output["timestamp"], tz=timezone.utc
        ).isoformat()

    # Numeric status codes → human-readable strings
    status_map = {200: "active", 0: "inactive", 404: "not_found", 403: "suspended"}
    if "status" in tool_output and isinstance(tool_output["status"], int):
        tool_output["status"] = status_map.get(tool_output["status"], "unknown")

    return {
        "hookSpecificOutput": {
            "hookEventName": input_data["hook_event_name"],
            "additionalContext": f"Normalized output: {json.dumps(tool_output)}"
        }
    }
```

The architectural point is important: the hook centralizes normalization so the model consistently sees clean data, regardless of which tool produced it.

### Wiring hooks to options

Hooks are registered in `ClaudeAgentOptions` using `HookMatcher` objects:

```python
options = ClaudeAgentOptions(
    hooks={
        "PreToolUse": [
            HookMatcher(matcher="process_refund", hooks=[enforce_refund_policy])
        ],
        "PostToolUse": [
            HookMatcher(hooks=[normalize_tool_outputs])  # no matcher = all tools
        ]
    }
)
```

## Matchers keep hooks targeted

Use matchers to narrow hooks to the tools that matter:

| Pattern | Matches |
|---|---|
| `"Bash"` | Only the `Bash` tool |
| `"Write\|Edit"` | `Write` OR `Edit` |
| `"^mcp__"` | All MCP tools (starts with `mcp__`) |
| *(no matcher)* | All events of that hook type |

This matters on the exam because a targeted hook is usually cleaner than a global one.

## Hooks vs prompts: the real decision

The reason to use hooks instead of prompt instructions is **determinism**:

- **Hooks:** Deterministic. A `deny` decision *always* blocks the tool. No exceptions.
- **Prompts:** Probabilistic. Claude follows instructions most of the time, but not 100% of the time.

For security checks, financial operations, compliance rules, or irreversible actions, "usually" is not enough. That is the exam's key distinction.

Prompts are still great for preferences, tone, and soft guidance. Hooks are for hard guarantees.

- Task Statement 1.5 is mostly about `PreToolUse` and `PostToolUse`
- Use `PreToolUse` for blocking, prerequisite gates, and other hard controls
- Use `PostToolUse` for normalization before the model reasons over the result
- A `deny` decision blocks the tool and gives Claude actionable feedback
- Matchers let you apply hooks precisely instead of globally
- The docs include many other hook events, but they are not the core of this exam objective
