## What are hooks?

Hooks are functions that the Agent SDK calls at specific points during an agent's lifecycle. They let you intercept and react to events — before a tool runs, after it returns, when a session starts, when an agent goes idle, and more.

Think of hooks as middleware for agent actions. Just as web framework middleware can inspect, modify, block, or log HTTP requests before they reach your handler, hooks can inspect, modify, block, or log tool calls before they reach their execution logic.

This gives you a clean, centralized place to implement concerns that cut across all tool calls: security enforcement, data normalization, auditing, telemetry, and error handling — without cluttering your agent's system prompt or tool logic.

## How it works

```
Claude decides to call a tool
          │
          ▼
   ┌─────────────┐
   │ PreToolUse  │  ← Your hook fires here
   │   hooks     │    Can: allow, deny, modify input
   └──────┬──────┘
          │  (if allowed)
          ▼
   ┌─────────────┐
   │    Tool     │
   │  executes   │
   └──────┬──────┘
          │
          ▼
   ┌─────────────┐
   │ PostToolUse │  ← Your hook fires here
   │   hooks     │    Can: normalize output, audit, trigger side effects
   └──────┬──────┘
          │
          ▼
    Result returned to Claude
```

The `PreToolUse` hook fires before the tool executes. The `PostToolUse` hook fires after it returns. Between those two points, the tool runs normally.

## All hook types

| Hook | When it fires | Common uses |
|---|---|---|
| `PreToolUse` | Before a tool executes | Block dangerous commands, validate inputs, enforce prerequisites |
| `PostToolUse` | After a tool returns successfully | Normalize output data, audit trail, trigger side effects |
| `PostToolUseFailure` | After a tool fails | Log errors, alert on failures |
| `TeammateIdle` | When an agent teammate is about to go idle | Enforce quality gates before the agent stops |
| `UserPromptSubmit` | When a new prompt is submitted | Inject additional context, sanitize input |
| `Stop` | When the agent finishes | Save session state, validate final result |
| `SubagentStart` | When a subagent spawns | Track parallel tasks, initialize subagent telemetry |
| `SubagentStop` | When a subagent completes | Aggregate results, update progress tracking |
| `SessionStart` | When a session initializes | Inject environment context, load defaults, initialize telemetry |
| `PreCompact` | Before context compaction | Archive the full transcript before it's summarized |
| `Notification` | On agent status messages | Forward alerts to Slack, PagerDuty, or monitoring systems |

## In practice

### PreToolUse: blocking policy violations

The most common use of `PreToolUse` is enforcing business rules that must not be violated. Here's a hook that blocks refunds above an auto-approval threshold:

```python
async def enforce_refund_policy(input_data, tool_use_id, context):
    """Block refunds exceeding $500 auto-approval threshold."""
    if input_data["tool_name"] == "process_refund":
        amount = input_data["tool_input"].get("amount", 0)
        if amount > 500:
            return {
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "permissionDecision": "deny",
                    "permissionDecisionReason": (
                        f"Refund of ${amount} exceeds $500 limit. Escalate to human agent."
                    )
                },
                "systemMessage": "This refund requires human approval. Use the escalate_to_human tool."
            }
    return {}
```

When this hook returns a `deny` decision, the tool does not execute. Claude receives the `permissionDecisionReason` as feedback and can adapt — in this case, by escalating to a human instead.

The `systemMessage` field injects additional guidance into Claude's context, steering it toward the correct next action.

### PostToolUse: normalizing heterogeneous data

When your agent works with multiple external tools or APIs, they often return data in incompatible formats — different date representations, status codes as numbers vs strings, varying field names. A `PostToolUse` hook is the right place to normalize all of this before Claude sees it:

```python
async def normalize_tool_outputs(input_data, tool_use_id, context):
    """Normalize date formats and status codes from different MCP tools."""
    tool_output = input_data.get("tool_output", {})

    # Unix timestamp → ISO 8601
    if "timestamp" in tool_output and isinstance(tool_output["timestamp"], int):
        from datetime import datetime, timezone
        tool_output["timestamp"] = datetime.fromtimestamp(
            tool_output["timestamp"], tz=timezone.utc
        ).isoformat()

    # Numeric status codes → human-readable strings
    status_map = {200: "active", 0: "inactive", 404: "not_found", 403: "suspended"}
    if "status" in tool_output and isinstance(tool_output["status"], int):
        tool_output["status"] = status_map.get(tool_output["status"], "unknown")

    return {
        "hookSpecificOutput": {
            "hookEventName": "PostToolUse",
            "additionalContext": f"Normalized output: {json.dumps(tool_output)}"
        }
    }
```

By normalizing in a hook, you keep this logic in one place. Claude always sees consistent data formats regardless of which tool produced the output.

### Wiring hooks to options

Hooks are registered in `ClaudeAgentOptions` using `HookMatcher` objects that pair matchers with hook functions:

```python
options = ClaudeAgentOptions(
    hooks={
        "PreToolUse": [
            HookMatcher(matcher="process_refund", hooks=[enforce_refund_policy])
        ],
        "PostToolUse": [
            HookMatcher(hooks=[normalize_tool_outputs])  # no matcher = all tools
        ]
    }
)
```

## Matcher syntax

Matchers let you target hooks precisely — running them only for specific tools rather than every tool call:

| Pattern | Matches |
|---|---|
| `"Bash"` | Only the `Bash` tool |
| `"Write\|Edit"` | `Write` OR `Edit` |
| `"^mcp__"` | All MCP tools (starts with `mcp__`) |
| *(no matcher)* | All events of that hook type |

Targeted matchers are more efficient than running every hook on every tool call, and they make your hook logic easier to reason about — each hook only handles what it's designed for.

## Hooks vs prompts: the reliability gap

The fundamental reason to use hooks instead of (or in addition to) prompt instructions is reliability:

- **Hooks:** Deterministic. A `deny` decision *always* blocks the tool. No exceptions.
- **Prompts:** Probabilistic. Claude follows instructions most of the time, but not 100% of the time.

For security checks, financial operations, compliance requirements, and irreversible actions, "most of the time" is not sufficient. Hooks close that gap.

This doesn't mean prompts are useless — they're excellent for style, preferences, and soft guidelines. The skill is knowing which tool to reach for when the stakes are different.

## Key things to know

- `PreToolUse` fires before execution — use it for blocking, validation, and prerequisite gates
- `PostToolUse` fires after execution — use it for normalization, auditing, and side effects
- A `deny` decision blocks the tool and sends `permissionDecisionReason` to Claude as feedback
- The `systemMessage` field in a hook response injects guidance into Claude's context
- Matcher syntax allows precise targeting: exact names, regex patterns, or no matcher for all tools
- `TeammateIdle` enforces quality gates before an agent teammate stops — useful in multi-agent workflows
- `SessionStart` is the right place to inject environment context and initialize telemetry
- `PreCompact` fires before context compaction — use it to archive full transcripts

## Exam tips

- Questions may give you a scenario and ask which hook type to use — `PreToolUse` for prevention/blocking, `PostToolUse` for transformation/auditing after the fact.
- The deny decision is the key mechanism: it's not a soft suggestion to Claude, it's a hard block on execution.
- Matcher syntax questions: `"^mcp__"` matches all MCP tools, a plain tool name matches only that tool, no matcher matches everything.
- `TeammateIdle` is easy to overlook — remember it fires when an agent teammate is about to go idle, making it useful for quality gates in multi-agent systems.
- Don't confuse `SubagentStart`/`SubagentStop` (subagent lifecycle) with `SessionStart`/`Stop` (overall session lifecycle).
