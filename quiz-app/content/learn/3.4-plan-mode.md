## What is plan mode?

When Claude Code runs in its default mode, it can read files, make edits, run commands — all in one go. For a simple bug fix, that's exactly what you want. But for a large architectural change affecting dozens of files, executing immediately without a plan can cause costly rework when unexpected dependencies surface mid-implementation.

**Plan mode** (`permission_mode: "plan"`) puts Claude into a read-only exploration phase. It can read files, run searches, and reason about the codebase — but it cannot write, edit, or execute commands. The output is a plan for your review before any changes are made.

## How it works

```
permission_mode: "plan"        permission_mode: "default"
─────────────────────          ──────────────────────────
Claude can: Read               Claude can: Read
            Glob                           Glob
            Grep                           Grep
            Bash (read-only)               Bash (full)
                                           Edit
Claude cannot: Edit                        Write
               Write
               Bash (writes/execs)
```

Setting plan mode in code:

```python
options = ClaudeAgentOptions(
    permission_mode="plan"  # No writes — Claude produces a plan for review
)
```

You can also activate plan mode interactively: press **Shift+Tab** to toggle between plan and execute modes in the Claude Code REPL.

## When to use plan mode vs direct execution

| Situation | Use |
|---|---|
| Large-scale refactor (e.g., monolith → microservices) | Plan mode |
| Library migration affecting 40+ files | Plan mode |
| Choosing between integration approaches with infrastructure trade-offs | Plan mode |
| Multi-file changes where the right approach isn't obvious | Plan mode |
| Single-file bug fix with a clear stack trace | Direct execution |
| Adding a validated conditional | Direct execution |
| Well-understood change with clear scope | Direct execution |

The guiding question: *Would discovering an unexpected dependency mid-implementation cause expensive rework?* If yes, plan first.

## The Explore subagent for verbose discovery

When the discovery phase of a task will produce verbose output — reading hundreds of files, running many searches — the Explore subagent isolates that verbosity. The main agent receives a concise summary, not 50,000 tokens of file contents:

```python
# Coordinator spawns an Explore subagent for codebase mapping
# Explore reads files, runs Grep, follows imports
# Main agent receives: "Found 45 service files in src/services/. Key dependency: auth.ts..."
# Not: the full content of all 45 files
```

## Plan → Execute workflow

For complex changes, plan mode and direct execution work best in sequence:

```
Phase 1 — Plan mode
  └─ Explore codebase
  └─ Understand dependencies
  └─ Produce proposed approach for human review

Phase 2 — Direct execution
  └─ Human approves the plan
  └─ Claude implements the planned approach
```

This is also how CLAUDE.md `paths:` rules (lesson 3.3) interact with plan mode: when Claude reads a matching file during the exploration phase, the relevant rule loads automatically.

## Key things to know

- `permission_mode: "plan"` allows reads but blocks writes, edits, and shell execution
- Plan mode is for complex, multi-file changes where approach uncertainty would cause rework
- Direct execution is fine for clear, well-scoped changes
- The Explore subagent keeps verbose discovery output out of the main agent's context
- Plan mode ≠ safe mode — Claude still reads files and uses tools; it just can't change anything

## Exam tips

- Plan mode does not mean "Claude thinks before acting" — it means `permission_mode: "plan"` blocks write/edit/exec tools
- A question about "reviewing a proposed plan before Claude makes changes" → plan mode, not a prompt instruction to "think first"
- The Explore subagent is distinct from plan mode — it's a subagent type optimised for codebase discovery, not a permission restriction
- "Shift+Tab toggles plan mode" is a real exam fact about the interactive REPL
