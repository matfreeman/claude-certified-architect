## What is the CLAUDE.md hierarchy?

CLAUDE.md files are plain text (Markdown) files that tell Claude Code how to behave in a project or on a machine. They contain project conventions, coding standards, build commands, team norms — anything you'd otherwise have to repeat in every prompt.

The hierarchy exists because different instruction sets have different audiences. Your organisation might have security policies that must apply everywhere. Your project might have conventions that all contributors need. You personally might have preferences that only affect your own machine. The hierarchy lets all of these coexist without stepping on each other.

The key thing to understand is that these files are **concatenated, not overriding**. All applicable CLAUDE.md files are loaded and merged into Claude's context. "Most specific wins" applies when instructions genuinely conflict, but in practice the files are additive.

## How it works

There are four levels, from broadest to most personal:

```
Level 1: Managed policy      /Library/Application Support/ClaudeCode/CLAUDE.md  (macOS)
Level 2: Project             ./CLAUDE.md  or  ./.claude/CLAUDE.md
Level 3: Local               ./CLAUDE.local.md
Level 4: User                ~/.claude/CLAUDE.md
```

```
┌─────────────────────────────────────────────────────────────┐
│  Managed policy  ← IT/MDM controlled, applies to all users  │
├─────────────────────────────────────────────────────────────┤
│  Project CLAUDE.md  ← committed to git, shared with team    │
├─────────────────────────────────────────────────────────────┤
│  CLAUDE.local.md  ← git-ignored, your personal notes        │
├─────────────────────────────────────────────────────────────┤
│  ~/.claude/CLAUDE.md  ← your preferences, all projects      │
└─────────────────────────────────────────────────────────────┘
```

At session start, Claude loads:
- All CLAUDE.md files found in directories **above** the current working directory
- The project-root CLAUDE.md
- The user-level CLAUDE.md

Files in **subdirectories** load on demand — when Claude reads a file in that directory, it also picks up any CLAUDE.md living there.

## The @import syntax

Large projects can split instructions across multiple files and pull them together with `@path` imports:

```markdown
# packages/api/CLAUDE.md

See @README for project overview.

## Standards
@../../docs/api-conventions.md
@../../docs/error-handling-standards.md
@../../docs/testing-conventions.md
```

This keeps individual CLAUDE.md files short while still making all relevant context available. The `@` path is resolved relative to the file containing it.

If your repo already uses `AGENTS.md` for another AI tool, create a CLAUDE.md that imports it and then adds Claude-specific guidance:

```markdown
@AGENTS.md

## Claude Code specifics

Use plan mode for changes under `src/billing/`.
```

## Organising with .claude/rules/

For larger projects, rules can be split into topic files under `.claude/rules/`:

```
.claude/
├── CLAUDE.md              ← short universal instructions
└── rules/
    ├── testing.md         ← testing conventions
    ├── api-conventions.md ← API design rules
    ├── security.md        ← security requirements
    └── deployment.md      ← deployment procedures
```

Rules files **without** `paths:` frontmatter load at launch alongside CLAUDE.md. Rules files **with** `paths:` frontmatter load conditionally when Claude works in matching paths (covered in detail in lesson 3.3).

## Auto memory and the /memory command

Claude Code can maintain persistent notes about your project in a `MEMORY.md` file (the auto memory system). This file lives in the project's `.claude/` directory and accumulates facts Claude has learned about your codebase across sessions.

- Auto memory is machine-local and project-scoped — it doesn't sync with teammates
- `MEMORY.md` is the entrypoint; detailed notes are written into topic files and referenced from there
- Only the beginning of `MEMORY.md` loads automatically at session start; detailed notes are read on demand

The `/memory` command shows you exactly which memory files are currently loaded in your session. If Claude seems to be ignoring instructions, run `/memory` first — if the file isn't listed, Claude can't see it.

## What survives /compact

When the conversation gets long and you run `/compact`, Claude summarises the session. Not everything survives:

| Content | After /compact |
|---|---|
| System prompt and output style | Unchanged |
| Project-root CLAUDE.md and unscoped rules | Re-injected from disk |
| Auto memory | Re-injected from disk |
| Rules with `paths:` frontmatter | Lost until a matching file is read again |
| Nested CLAUDE.md files in subdirectories | Lost until a file in that subdirectory is read again |
| Invoked skill bodies | Re-injected with token caps |

## Writing effective CLAUDE.md files

- Target under 200 lines per file — longer files reduce adherence
- Specific instructions work better than vague ones: `"Use 2-space indentation"` not `"Format code well"`
- Avoid contradictions across files — Claude may pick one arbitrarily
- HTML comments (`<!-- like this -->`) are stripped before injection — use them for maintainer notes without affecting Claude

## 📖 Read the official docs

**[Memory and CLAUDE.md docs](https://code.claude.com/docs/en/memory)** — Focus on the loading order table, the `.claude/rules/` section, and the auto memory behaviour. The `--add-dir` flag and `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1` environment variable are worth noting for completeness.

## Key things to know

- CLAUDE.md files are **concatenated**, not overriding — all applicable files load and merge
- Files in subdirectories load **on demand** when Claude reads files in those directories
- `~/.claude/CLAUDE.md` is personal and **not shared** via version control — teammates won't see it
- `CLAUDE.local.md` loads after `CLAUDE.md` at the same level — personal overrides at the project level
- The `/memory` command is your diagnostic tool: if a file isn't listed, Claude doesn't have it
- `--append-system-prompt` is a different mechanism from CLAUDE.md — it injects directly into the system prompt for one session
- `--bare` skips CLAUDE.md auto-discovery entirely, useful for reproducible CI runs

## Exam tips

- If a new team member isn't getting project instructions, the likely cause is that instructions are in `~/.claude/CLAUDE.md` (user-level) instead of the project-level file — user-level is never shared via git
- Rules without `paths:` frontmatter in `.claude/rules/` load at launch; rules with `paths:` frontmatter load conditionally — don't confuse the two
- `/compact` re-injects CLAUDE.md from disk, but path-scoped rules and subdirectory CLAUDE.md files are lost until the relevant files are accessed again
- CLAUDE.md is injected as context **after** the system prompt, not as part of the system prompt itself
- When a repo uses `AGENTS.md`, the correct pattern is to import it via `@AGENTS.md` rather than duplicating its contents
