## What are validation and retry loops?

JSON schemas (lesson 4.3) guarantee that Claude's output has the right *structure*. But they can't catch *semantic* errors — a total_amount field that doesn't match the sum of line items, or a payment_terms of "other" with no detail provided.

Validation and retry loops close this gap. After each extraction, you run application-level checks. If they fail, you return the specific errors to Claude as tool results and ask it to correct the output. Claude sees its own output alongside the failure reason and can usually fix it on the next attempt.

## How it works

```
┌─────────────────────────────────────────────────────┐
│                   Retry loop                        │
│                                                     │
│  1. Call API with extraction tool                   │
│  2. Extract result from tool_use block              │
│  3. Run semantic validation                         │
│  4. If errors → append to messages, call again      │
│  5. Repeat up to max_retries                        │
└─────────────────────────────────────────────────────┘
```

```python
def extract_with_retry(document: str, max_retries: int = 3) -> dict:
    messages = [{"role": "user", "content": document}]
    
    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=[extraction_tool],
            tool_choice={"type": "tool", "name": "extract_invoice_data"},
            messages=messages
        )
        
        for block in response.content:
            if block.type == "tool_use":
                extracted = block.input
                errors = validate_extraction(extracted)
                
                if not errors:
                    return extracted  # Success
                
                # Retry with specific error feedback
                messages.extend([
                    {"role": "assistant", "content": response.content},
                    {
                        "role": "user",
                        "content": [{
                            "type": "tool_result",
                            "tool_use_id": block.id,
                            "content": f"Validation failed. Please correct:\n{chr(10).join(errors)}\n\nOriginal document:\n{document}"
                        }]
                    }
                ])
    
    raise Exception(f"Extraction failed after {max_retries} attempts")
```

## Semantic validation — what schemas can't check

```python
def validate_extraction(data: dict) -> list[str]:
    errors = []
    
    # Line items must sum to total
    if data.get("line_items") and data.get("total_amount"):
        calculated = sum(item["amount"] for item in data["line_items"])
        if abs(calculated - data["total_amount"]) > 0.01:
            errors.append(f"Line items sum to {calculated} but stated total is {data['total_amount']}")
    
    # "other" requires detail
    if data.get("payment_terms") == "other" and not data.get("payment_terms_detail"):
        errors.append("payment_terms is 'other' but payment_terms_detail is missing")
    
    return errors
```

## When retries will and won't work

| Situation | Retry effective? |
|---|---|
| Wrong date format ("Jan 15" instead of "2025-01-15") | ✅ Formatting instructions fix this |
| Field placed in wrong schema slot | ✅ Specific error message fixes this |
| Information not present in the source document | ❌ Claude can't fabricate data that doesn't exist |
| External document referenced but not provided | ❌ Claude can't access what's not in context |

Retries fix *mistakes*, not *missing information*. If the data genuinely isn't in the document, no number of retries will help.

## Self-correction with calculated fields

Adding a calculated field alongside a stated field lets the model flag its own inconsistencies:

```python
# Add both stated and calculated totals to the schema
{
    "stated_total": 1500.00,       # From the document
    "calculated_total": 1450.00,   # Sum of line_items (model computes this)
    "total_conflict_detected": True # Model flags the discrepancy
}
```

Your validation code then routes conflicting extractions to human review rather than accepting them silently.

## The `detected_pattern` field for false positive analysis

When reviewing code or documents over time, capturing *why* each finding was flagged helps you improve your criteria:

```python
{
    "issue": "Unparameterized SQL query",
    "detected_pattern": "string concatenation in cursor.execute()",
    "severity": "critical"
}
```

If developers repeatedly dismiss findings with a particular `detected_pattern`, it may indicate a false positive category specific to your codebase — and you have the data to prove it.

## Key things to know

- Schemas validate structure; application code validates semantics
- Retry with specific error feedback is effective for formatting and structural mistakes
- Retries cannot fix absent information — the data must exist in the source document
- Three retries is a reasonable default; more than that suggests a systematic prompt problem
- The `detected_pattern` field is an investment: it makes false positive analysis tractable at scale

## Exam tips

- "Line items don't add up to the stated total" → semantic validation + retry, not schema changes
- "Retry loop never converges" → likely the information is absent from the document, not a prompt error
- Tool result messages must include `tool_use_id` — this is how Claude correlates results to its prior call
- Self-correction fields (calculated vs stated) are a detection pattern, not a guarantee — a human routing step still needed for conflicts
