## What are custom slash commands?

Claude Code lets you define your own `/commands` that appear in the command palette. You type `/review` and Claude runs whatever procedure you've written — no need to repeat long prompts or remember which flags to use.

A simple command is just a Markdown file. A skill is a command with extra capabilities: it can declare its own tool permissions, run in an isolated context, and even be invoked automatically by Claude when it recognises the situation.

This distinction matters for the exam: `.claude/commands/` files and `.claude/skills/<name>/SKILL.md` files both create slash commands. Skills just have more frontmatter options.

## How it works: scope and location

Commands and skills can be scoped to your project (shared with teammates) or to yourself (personal only):

```
Project scope                          Personal scope
──────────────────────────────────     ────────────────────────────────
.claude/commands/review.md             ~/.claude/commands/review.md
.claude/skills/review/SKILL.md         ~/.claude/skills/review/SKILL.md
  (version-controlled, shared)           (local only, not shared)
```

Project-scoped commands are available to everyone who clones the repository. Personal commands stay on your machine.

## SKILL.md frontmatter

A SKILL.md file is a Markdown file with YAML frontmatter. Here is a full example:

```yaml
---
name: review-pr
description: Reviews a pull request for security, quality, and test coverage. Use when asked to review a PR or check code quality.
argument-hint: [PR number or diff]
context: fork
allowed-tools: Read Grep Glob Bash(gh pr *)
disable-model-invocation: false
---

Review the pull request $ARGUMENTS:

1. Check for security vulnerabilities
2. Assess code quality and maintainability  
3. Verify test coverage is adequate
4. Check for documentation gaps

Format findings as: location | issue | severity | suggested fix
```

The frontmatter fields, explained:

| Field | What it does |
|---|---|
| `description` | The plain-language description Claude reads to decide whether to auto-invoke this skill |
| `argument-hint` | Shown during tab completion — tells the user what input is expected |
| `context: fork` | Runs the skill in an isolated subagent, so verbose output doesn't pollute the main conversation |
| `allowed-tools` | Tools pre-approved for this skill — no permission prompts during execution |
| `disable-model-invocation: true` | Prevents Claude from auto-invoking — only you can trigger it manually |
| `user-invocable: false` | Hides the skill from the `/` menu but lets Claude invoke it automatically |

## context: fork — why it matters

Without `context: fork`, a skill's output (all the files it reads, searches it runs, intermediate reasoning) flows into your main conversation context. For a quick command that's fine. For a skill that reads 50 files or runs a deep research pass, you don't want all that in your context window.

`context: fork` runs the skill in a subagent. The main conversation receives only the final output — a clean summary, not the full transcript.

Use it for:
- Verbose output skills (codebase analysis, deep research)
- Exploratory skills where divergent thinking shouldn't pollute the session
- Skills with side effects where intermediate steps are noise

## Skills vs CLAUDE.md

A common point of confusion: when should something be a skill vs in CLAUDE.md?

| Use CLAUDE.md | Use a skill |
|---|---|
| Always-on conventions (code style, build commands) | On-demand procedures (`/review`, `/deploy`) |
| Short universal standards | Verbose reference material (style guide with 200 examples) |
| Things that should apply to every interaction | Things that apply only in specific situations |

CLAUDE.md loads unconditionally. Skills load only when invoked. If you put a 500-line style guide in CLAUDE.md, it consumes context on every single turn. As a skill, it only uses context when someone types `/style-guide`.

## Creating a shared /review command

To make `/review` available to all developers after `git clone`:

```
.claude/
└── commands/
    └── review.md
```

Or equivalently:

```
.claude/
└── skills/
    └── review/
        └── SKILL.md
```

Both approaches create the `/review` command and commit to version control.

To create a personal variant without affecting teammates, create it at `~/.claude/skills/review-strict/SKILL.md` with a different name. Personal skills don't shadow project skills unless they share the exact same name.

## Key things to know

- `.claude/commands/` and `.claude/skills/` both create slash commands — skills add frontmatter superpowers
- `context: fork` runs the skill in an isolated subagent — the main context receives only the result
- The `description` field is what enables Claude to auto-invoke a skill — write it clearly
- `disable-model-invocation: true` makes the skill user-only; `user-invocable: false` makes it Claude-only
- Project-scoped skills are version-controlled and shared; user-scoped skills are personal
- CLAUDE.md is for always-on instructions; skills are for on-demand procedures

## Exam tips

- If a question asks how to make a command available to all developers without manual setup, the answer is a project-scoped command in `.claude/commands/` or `.claude/skills/` — both are version-controlled
- `context: fork` isolates skill output from the main conversation — it doesn't restrict what tools the skill can use
- `disable-model-invocation: true` prevents Claude from triggering the skill automatically; without it, a descriptive `description` field may cause Claude to invoke it on its own
- The `description` field drives auto-invocation — a vague description means Claude won't know when to use the skill
- Personal skills at `~/.claude/skills/` only conflict with project skills if they share the same name
