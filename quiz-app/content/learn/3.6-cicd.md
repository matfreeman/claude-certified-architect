## What is CI/CD integration for Claude Code?

Claude Code is typically an interactive tool — you type a prompt, it reads files, asks questions, makes changes. But in a CI/CD pipeline (GitHub Actions, GitLab CI, Jenkins), there's no human present. Claude needs to run non-interactively: start, complete the task, exit.

This is called **headless mode**. The same Claude Code binary runs, but with flags that prevent it from waiting for input and ensure its output is machine-parseable.

## How headless mode works

The single most important flag is `-p` (or `--print`):

```bash
# WRONG — hangs waiting for interactive input
claude "Review this PR for security issues"

# CORRECT — non-interactive, exits after completing
claude -p "Review this PR for security issues"

# Pre-approve specific tools so Claude doesn't stop to ask
claude -p "Run the test suite and fix failures" --allowedTools "Bash,Read,Edit"
```

Without `-p`, Claude Code enters interactive mode and waits. In a CI job, this causes the job to hang indefinitely.

## Structured JSON output

For automated workflows — posting PR comments, generating reports, triggering downstream jobs — you need Claude's output in a consistent machine-readable format:

```bash
claude -p "Analyze this PR for security vulnerabilities" \
  --output-format json \
  --json-schema '{
    "type": "object",
    "properties": {
      "findings": {
        "type": "array",
        "items": {
          "properties": {
            "file":           {"type": "string"},
            "line":           {"type": "integer"},
            "severity":       {"type": "string", "enum": ["critical", "high", "medium", "low"]},
            "issue":          {"type": "string"},
            "suggested_fix":  {"type": "string"}
          }
        }
      }
    }
  }'
```

The response is wrapped in a JSON object; your schema-conformant result is in the `structured_output` field. Parse it with `jq`:

```bash
| jq '.structured_output.findings[] | select(.severity == "critical")'
```

## CLAUDE.md provides context in CI

Your project's `CLAUDE.md` is still loaded when Claude runs in CI. This is how you give the CI instance the same project context as your local development environment — testing standards, review criteria, available fixtures:

```markdown
# CI Review Instructions

## Testing Standards
- Unit tests required for all business logic
- Integration tests for API endpoints

## Review Criteria
- Flag: SQL injection, XSS, unvalidated input
- Flag: Missing error handling in API handlers
- Skip: style issues, minor naming conventions
```

## Avoiding duplicate comments and suggestions

If Claude runs on every commit, it will re-report the same findings unless you include prior context:

```bash
# Provide previous findings so Claude only reports new issues
claude -p "Review the new changes in this PR.

Prior review findings (already commented):
$(cat previous-findings.json)

Only report issues that are NEW or still unaddressed." \
  --output-format json
```

Same principle for test generation — give Claude the existing test file so it doesn't suggest tests that already exist.

## The `--bare` flag

`--bare` skips auto-discovery of hooks, skills, MCP servers, and CLAUDE.md entirely:

```bash
claude --bare -p "Summarize this file" --allowedTools "Read"
```

Use `--bare` in CI when you want completely reproducible, environment-independent behaviour — no local configuration can affect the run.

## Session isolation for reviews

A Claude session that generated code retains the reasoning context from generation — it's less likely to question decisions it just made. For code review in CI, always use a fresh session (not the session that wrote the code):

```
Generate code  →  Session A
Review code    →  Session B (independent — no generation context)
```

## Key things to know

- `-p` / `--print` is required for any non-interactive invocation — without it, Claude hangs
- `--output-format json` + `--json-schema` gives machine-parseable structured output
- The project `CLAUDE.md` still loads in CI unless you pass `--bare`
- `--bare` skips all auto-discovery — use it for reproducible CI runs unaffected by local config
- Session isolation prevents the code generator from reviewing its own output (it retains bias toward its own decisions)
- GitHub Actions and GitLab CI both reduce to the same pattern: non-interactive invocation with explicit tool permissions

## Exam tips

- "Claude hangs in CI" → missing `-p` flag (not a permissions issue, not a timeout)
- "`--bare` skips CLAUDE.md" — true. This is distinct from `--allowedTools`, which controls tool permissions
- Session isolation for review is about *reasoning context*, not permissions — a fresh session has no prior bias
- `--output-format json` returns the schema result in `structured_output`, not at the top level of the response
