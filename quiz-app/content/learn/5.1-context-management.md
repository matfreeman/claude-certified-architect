## What is conversation context management?

Every Claude API call sends the *entire conversation history* — every message, every tool call, every tool result — from scratch. As a conversation grows, this accumulation has two effects: you approach the context window limit, and Claude's attention quality degrades for information buried in the middle.

Context management is the discipline of controlling *what* goes into the context window, *where* it's positioned, and *how verbose* it is. Done well, it keeps long-running agents functional and accurate. Done poorly, your agent starts losing track of facts it saw 20 turns ago.

## What accumulates in context

```
[ system prompt ]
+ [ CLAUDE.md / rules ]
+ [ tool definitions ]
+ [ every conversation turn — both sides ]
+ [ every tool input ]
+ [ every tool output ]
= total context consumed
```

Tool outputs are often the fastest-growing element. A `lookup_order` call that returns a 40-field object contributes all 40 fields — even if you only need 5.

## The "lost in the middle" effect

Claude reliably processes information at the **beginning** and **end** of long inputs. Information in the middle sections receives less attention. For a conversation that runs hundreds of turns, anything from the middle of the history may effectively be invisible.

```
[ Beginning — high attention ]
   ... many turns ...
[ Middle — reduced attention ]
   ... more turns ...
[ End — high attention ]
```

Critical facts belong at the start or end, never buried in the middle.

## The persistent "case facts" block

For long-running workflows (customer support, multi-step investigations), extract transactional facts into a structured block that sits at the *beginning* of every prompt — outside the summarized history:

```python
case_facts = {
    "customer_id": "C-12345",
    "issue_type": "billing_dispute",
    "disputed_amount": 150.00,
    "customer_stated_expectation": "Full refund within 48 hours",
    "case_opened": "2025-04-15T14:30:00Z"
}

prompt = f"""
## Active Case Facts (preserve verbatim — do not summarize)
{json.dumps(case_facts, indent=2)}

## Conversation Summary
{summarized_history}

## Current Message
{current_message}
"""
```

The case facts block never gets compressed into the summary. It sits at the top of every call, immune to progressive summarization loss.

## Progressive summarization risks

When you summarize a long conversation to save context space, the summary may lose:
- Specific numbers (amounts, order numbers, percentages)
- Exact dates and deadlines
- Customer-stated expectations ("I need this fixed by Friday")
- Technical specifications

Summaries compress. Facts block don't. Keep critical specifics in the facts block, not in the summarized history.

## Trimming tool outputs

```python
def trim_order_lookup(full_result: dict) -> dict:
    """Keep only return-relevant fields from a 40-field order object."""
    return {
        "order_id": full_result["order_id"],
        "order_date": full_result["order_date"],
        "status": full_result["status"],
        "total_amount": full_result["total_amount"],
        "items": full_result["items"],
        "return_eligibility": full_result["return_eligibility"],
        # Dropped: shipping address, payment method, internal SKUs, warehouse codes
    }
```

Trim before appending to the conversation. The full result can live in your database — Claude only needs the relevant fields.

## Position-aware input ordering for multi-agent synthesis

When a downstream agent receives aggregated findings from multiple subagents, position the most important content at the beginning and end:

```python
prompt = f"""
## Key Findings Summary
{key_findings_summary}      # ← BEGINNING — gets full attention

## Detailed Results: Section A
{section_a_results}

## Detailed Results: Section B
{section_b_results}

## Integration Notes
{integration_notes}         # ← END — also gets good attention
"""
```

## Key things to know

- Claude sends the full conversation history on every API call — history accumulates
- The "lost in the middle" effect means critical facts need to be at the start or end
- A persistent case facts block (not in the summary) preserves specific numbers and dates
- Trim tool outputs before appending them — send only the fields Claude needs
- Progressive summarization compresses facts; the case facts block preserves them verbatim

## Exam tips

- "Agent forgot a fact from earlier in the conversation" → lost in the middle effect + position-aware ordering
- "Customer stated expectation disappeared after /compact" → not in case facts block (it was in the summarized history)
- The fix for verbose tool outputs is trimming before appending, not a different tool
- Upstream agents should return structured findings, not verbose reasoning chains, to help downstream context budgets
