### 1.7 Session State, Resumption, and Forking

## What a session actually is

A **session** is the conversation history the SDK accumulates while an agent works:

- your prompt
- every tool call
- every tool result
- every model response

The SDK persists that transcript to disk automatically, so you can come back to it later with full prior context.

That means a resumed session remembers:

- files it already read
- analysis it already performed
- decisions it already made

But sessions persist the **conversation**, not the **filesystem**. If the agent edited files, resuming the session does not give you an isolated copy of those edits. For filesystem branching or rollback, use file checkpointing.

## When session management matters

You only need to think about sessions when you send **multiple prompts that should share context**.

Within a single `query()` call, the agent already handles its own internal turns. Permission prompts and `AskUserQuestion` are handled **in-loop**, so they do not end the call and do not require manual resume logic.

## Choose the right approach

| Situation | What to use | Why |
|---|---|---|
| One-shot task, no follow-up | One `query()` call | Nothing extra needed |
| Multi-turn conversation in one process | Python `ClaudeSDKClient` or TypeScript `continue: true` | Lets the SDK keep using the same conversation |
| Restart process and continue the latest session | Python `continue_conversation=True` or TypeScript `continue: true` | Reopens the most recent session in the current directory |
| Return to a specific old session | `resume=session_id` / `resume: sessionId` | Needed when you track multiple sessions |
| Try an alternative approach without losing the original | `fork_session=True` / `forkSession: true` | Creates a new branch of history |
| Stateless task with no disk persistence | TypeScript `persistSession: false` | Session lives only in memory for that call |

Python always persists sessions to disk. The docs call out `persistSession: false` as a TypeScript-only option.

## Continue vs resume vs fork

These three are easy to confuse, so keep the distinction sharp:

- **Continue**: find the **most recent** session in the current directory automatically
- **Resume**: reopen one **specific** session by ID
- **Fork**: create a **new** session that starts from a copy of an existing session's history

Continue and resume both add new turns to an existing conversation. Fork creates a second conversation history so you can explore a different direction without overwriting the original.

## Automatic session management

### Python: `ClaudeSDKClient`

Use `ClaudeSDKClient` for multi-turn conversations inside one running process. The client tracks the session ID internally, so each `client.query()` continues the same session automatically.

```python
import asyncio
from claude_agent_sdk import (
    ClaudeSDKClient,
    ClaudeAgentOptions,
    AssistantMessage,
    ResultMessage,
    TextBlock,
)


def print_response(message):
    if isinstance(message, AssistantMessage):
        for block in message.content:
            if isinstance(block, TextBlock):
                print(block.text)
    elif isinstance(message, ResultMessage):
        print(f"[done: {message.subtype}]")


async def main():
    options = ClaudeAgentOptions(
        allowed_tools=["Read", "Edit", "Glob", "Grep"],
    )

    async with ClaudeSDKClient(options=options) as client:
        await client.query("Analyze the auth module")
        async for message in client.receive_response():
            print_response(message)

        await client.query("Now refactor it to use JWT")
        async for message in client.receive_response():
            print_response(message)


asyncio.run(main())
```

### TypeScript: `continue: true`

The stable TypeScript SDK does not use a session-holding client object like Python's `ClaudeSDKClient`. Instead, you make another `query()` call with `continue: true`, and the SDK resumes the most recent session on disk in the current directory.

## Manual session options with `query()`

### Capture the session ID

If you want to resume a specific prior session or fork one later, capture the `session_id` from the result message. The docs note that it is present on every result message, whether the run ended in success or error.

```python
from claude_agent_sdk import query, ClaudeAgentOptions, ResultMessage


async def capture_session_id():
    session_id = None

    async for message in query(
        prompt="Analyze the auth module and suggest improvements",
        options=ClaudeAgentOptions(
            allowed_tools=["Read", "Glob", "Grep"],
        ),
    ):
        if isinstance(message, ResultMessage):
            session_id = message.session_id
            if message.subtype == "success":
                print(message.result)

    print(f"Session ID: {session_id}")
    return session_id
```

### Resume by ID

Use `resume` when you need a specific prior conversation rather than "whatever was most recent."

Common reasons from the docs:

- follow up on a completed analysis without re-reading everything
- recover from `error_max_turns` or `error_max_budget_usd`
- restore a conversation after your process restarted

```python
async for message in query(
    prompt="Now implement the refactoring you suggested",
    options=ClaudeAgentOptions(
        resume=session_id,
        allowed_tools=["Read", "Edit", "Write", "Glob", "Grep"],
    ),
):
    if isinstance(message, ResultMessage) and message.subtype == "success":
        print(message.result)
```

### Fork to explore alternatives

Forking creates a new session that starts with a copy of the old session's history. The fork gets a **new** session ID, while the original session stays unchanged.

```python
forked_id = None

async for message in query(
    prompt="Instead of JWT, implement OAuth2 for the auth module",
    options=ClaudeAgentOptions(
        resume=session_id,
        fork_session=True,
    ),
):
    if isinstance(message, ResultMessage):
        forked_id = message.session_id
        if message.subtype == "success":
            print(message.result)

print(f"Forked session: {forked_id}")
```

The docs are explicit here: **forking branches conversation history, not the filesystem**. If the fork edits files, those edits are real and visible in the working directory. Use file checkpointing if you need file-level branch and revert behavior.

## Practical guidance from the docs

### If `resume` seems to start a fresh session

The most common cause is a mismatched `cwd`.

Sessions are stored under:

```text
~/.claude/projects/<encoded-cwd>/<session-id>.jsonl
```

The `<encoded-cwd>` is the absolute working directory with non-alphanumeric characters replaced by `-`.

So if you resume from a different directory, the SDK may look in the wrong place and fail to find the expected history.

### After resuming, tell the agent what changed

Even if the transcript resumes correctly, prior tool results can be stale if files changed since the earlier run. A good follow-up prompt makes that explicit:

```text
Since our last session, auth.py was refactored. The JWTManager class was split into
TokenIssuer and TokenValidator. Re-read auth.py before continuing.
```

### Resuming across hosts

Session files are local to the machine that created them. The docs give two main approaches:

- move the session file to the same path on the new host and keep `cwd` consistent
- avoid relying on transcript resume and instead pass the important analysis/results into a fresh session

For shared infrastructure or serverless setups, the docs point to a `SessionStore` adapter.

### Session utilities

The SDK also exposes functions for:

- listing sessions
- reading session messages
- getting session info
- renaming sessions
- tagging sessions

These are useful if you want to build session pickers, transcript viewers, cleanup jobs, or human-readable labels on top of the SDK.

## Key things to know

- A session stores conversation history, not a snapshot of the filesystem
- One `query()` call already handles multiple internal turns, approvals, and `AskUserQuestion` in-loop
- Use `continue` for the most recent session, `resume` for a specific session ID, and `fork` for a new branch of history
- Python `ClaudeSDKClient` is the easiest option for multi-turn conversations in one process
- Forking creates a new conversation history but does not isolate file edits
- If `resume` seems wrong, check `cwd` first
- Resuming across machines is possible, but often a fresh session with injected state is more robust

## Exam tips

- "Continue the latest conversation" and "resume a specific past conversation" are different requirements
- Forking is the right answer when you want to explore alternatives without losing the original reasoning path
- If a question involves edited files after the original run, expect the best answer to mention stale tool results or telling the agent what changed
- If a scenario assumes sessions preserve file state, that is a trap: sessions preserve transcript history, not filesystem state
