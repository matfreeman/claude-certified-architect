## What is subagent invocation?

Knowing that coordinators spawn subagents is one thing — understanding the *mechanics* of how that happens is another. In the Agent SDK, subagents are not spawned by your application code. They are spawned by the coordinator agent itself, using a special tool called the `Agent` tool.

When you define your coordinator's options, you register named subagent definitions. At runtime, the coordinator decides when and how to invoke them, just like it decides when to call `Read` or `Grep`. Your job is to define the subagents clearly enough that the coordinator can make good invocation decisions.

## How it works

```
Your code                    Coordinator Claude          Subagent Claude
─────────────────────────    ──────────────────────────  ─────────────────
query(prompt, options)  ──▶  Receives task
                             Decides to spawn agents
                             Emits Agent("web-search",   ──▶  Runs with its own
                               prompt=...)                    system prompt +
                             Emits Agent("doc-analyzer", ──▶  tools + the prompt
                               prompt=...)                    the coordinator
                                                              composed
                                                         ◀──  Returns result
                             Receives both results
                             Passes to synthesizer   ──▶  ...
                             ...
                        ◀──  ResultMessage
```

The coordinator emits `Agent` tool calls just like it would emit `Bash` or `Read` calls. The SDK intercepts those calls, spawns isolated agent sessions, and returns results back to the coordinator as tool results.

### Defining subagents with `AgentDefinition`

You register subagents in the coordinator's `options.agents` dictionary. Each entry is a name (used by the coordinator to refer to it) and an `AgentDefinition`:

```python
from claude_agent_sdk import query, ClaudeAgentOptions, AgentDefinition

async for message in query(
    prompt="Research the impact of AI on creative industries",
    options=ClaudeAgentOptions(
        allowed_tools=["Read", "Grep", "Glob", "Agent"],  # "Agent" is required!
        agents={
            "web-search": AgentDefinition(
                description="Searches the web for recent articles and news. Use this to find current information.",
                prompt="""You are a web research specialist. Search for relevant content and return:
- Claim text
- Source URL
- Publication date
- Relevance score (1-10)

Return structured JSON with these fields for each finding.""",
                tools=["WebSearch", "WebFetch"],
            ),
            "doc-analyzer": AgentDefinition(
                description="Analyzes documents and research papers in the local filesystem.",
                prompt="You are a document analysis specialist. Analyze the provided documents thoroughly and extract key findings with source citations.",
                tools=["Read", "Grep", "Glob"],
            ),
            "synthesizer": AgentDefinition(
                description="Synthesizes findings from multiple sources into a coherent report.",
                prompt="""You are a synthesis specialist. Combine provided findings into a coherent report.
Preserve all source attributions. Distinguish well-established findings from contested claims.""",
                tools=["Read"],
            ),
        },
    ),
):
    if hasattr(message, "result"):
        print(message.result)
```

Notice `"Agent"` in `allowed_tools`. Without it, the coordinator cannot spawn subagents. This is a common mistake — you set up beautiful `AgentDefinition` objects but forget to allow the `Agent` tool itself.

### `AgentDefinition` fields

| Field | Required | Description |
|---|---|---|
| `description` | Yes | When Claude should use this subagent. This is what the coordinator reads when deciding which agent to invoke. |
| `prompt` | Yes | The subagent's system prompt — its role, output format, and instructions. |
| `tools` | No | Tool list restricted to this subagent. If omitted, inherits all allowed tools. |
| `model` | No | `"sonnet"`, `"opus"`, `"haiku"`, or `"inherit"`. Use `"haiku"` for simple/cheap subagents. |
| `skills` | No | Named skills available to this subagent. |

The `description` field deserves special attention. The coordinator uses it to decide which agent to invoke for a given subtask — it's essentially the agent's "job posting." Vague descriptions lead to poor coordinator decisions. Specific, action-oriented descriptions ("Searches the web for recent articles and news. Use this to find current information.") lead to appropriate delegation.

## Passing context explicitly

Because subagents have isolated context (no access to the coordinator's conversation history), the coordinator must compose each subagent's prompt to include everything that subagent needs.

This is where many implementations go wrong. The coordinator might call the synthesizer with just "Synthesize the findings" — but the synthesizer has no idea what the findings are.

```python
# WRONG — synthesizer has no access to web_results or doc_analysis
subagent_prompt = "Analyze these documents and synthesize findings"

# CORRECT — include all findings in the prompt text
subagent_prompt = f"""
Synthesize these research findings into a comprehensive report:

## Web Search Results
{json.dumps(web_results, indent=2)}

## Document Analysis
{json.dumps(doc_analysis, indent=2)}

Requirements:
- Preserve source URLs for every claim
- Distinguish well-established from contested findings
- Include publication dates for temporal context
"""
```

The coordinator needs to be instructed (via its system prompt or the task prompt) to pass results this way. You can't assume it will do so by default.

### Preserving attribution metadata

When passing findings between agents, structure the data to carry attribution through the pipeline. A synthesizer that doesn't know where a claim came from can't produce a properly cited report:

```python
findings = {
    "claims": [
        {
            "claim": "AI tools reduced production time by 40%",
            "source_url": "https://example.com/article",
            "source_name": "Creative AI Report 2025",
            "publication_date": "2025-01-15",
            "excerpt": "...our study found that teams using AI tooling..."
        }
    ]
}
```

## Parallel spawning

Subagents that don't depend on each other should run in parallel. The coordinator achieves this by emitting multiple `Agent` tool calls in a single response — not across separate turns.

If the coordinator first spawns Web Search, waits for results, then spawns Document Analyzer, those agents run sequentially. If the coordinator spawns both in the same turn (emitting two `Agent` calls at once), they run concurrently.

You guide this behavior through the coordinator's system prompt:

```python
coordinator_system = """
When researching a broad topic:
1. Spawn the web-search and doc-analyzer subagents simultaneously in one response
2. Wait for both results before proceeding
3. Pass ALL findings to the synthesizer

Do NOT run agents sequentially — spawn them in parallel by emitting multiple Agent tool calls in one response.
"""
```

## A note on naming: `Agent` vs `Task`

In Claude Code v2.1.63, the subagent spawning tool was renamed from `"Task"` to `"Agent"`. Documentation and questions may reference either name — both refer to the same mechanism of spawning an isolated subagent session.

## Key things to know

- The coordinator spawns subagents by calling the `Agent` tool — `"Agent"` must be in `allowed_tools`
- Each subagent runs in an isolated session with only its `prompt` and the input the coordinator provides
- The `description` field is how the coordinator decides which agent to invoke for a given task
- Parallel spawning requires multiple `Agent` calls in a single coordinator turn, not across turns
- Subagents do not inherit parent context — pass everything explicitly in the prompt
- `AgentDefinition.model` lets you use cheaper models (`"haiku"`) for simple subagents
- The tool was formerly called `"Task"` — both names appear in documentation

## Exam tips

- Forgetting `"Agent"` in `allowed_tools` is a classic trap — the coordinator is fully configured but can't actually spawn subagents.
- Sequential vs parallel spawning: one `Agent` call per turn = sequential; multiple `Agent` calls in one turn = parallel.
- The `description` field drives the coordinator's agent-selection decisions — questions may test whether you know this vs. `prompt` (which drives the subagent's behavior once invoked).
- "Task" and "Agent" tool names both appear — don't be thrown off by either.
- Context isolation is absolute: if a subagent needs data, it must be in the prompt text, full stop.
