## What are structured MCP error responses?

When an MCP tool encounters a problem, it has two options: throw an exception that crashes the agent's execution, or return a structured error response that the agent can reason about and recover from. The second approach is almost always the right one.

Structured error responses let the agent — and Claude — understand what went wrong, decide whether recovery is possible, and take the appropriate next step. A bare exception terminates the workflow. A structured error opens up a recovery path.

The mechanism that enables this is the `isError` flag in the MCP tool response. When set to `true`, it signals that the tool call was attempted but failed. The rest of the response body carries the information the agent needs to decide what to do next.

## How it works

An MCP tool response is a JSON object with a `content` array and an optional `isError` flag. When `isError` is `false` (or absent), the agent treats the response as a successful result. When `isError` is `true`, the agent knows something went wrong — but the flag alone tells it nothing about what to do.

```
MCP Tool Response
├── isError: true/false
└── content: [{ type: "text", text: "...", ...metadata }]
                                              └── errorCategory, isRetryable, retryAfterMs, etc.
```

The metadata fields in the content object are what turn a bare failure signal into actionable intelligence.

### The four error categories

Each error category implies a different recovery strategy. Getting this right is the difference between an agent that recovers gracefully and one that retries indefinitely — or gives up too soon.

| Category | When to use it | What the agent should do |
|---|---|---|
| `transient` | Network timeout, service temporarily unavailable, rate limit hit | Retry with exponential backoff |
| `validation` | Invalid input format, missing required field, malformed request | Do not retry — fix the input first |
| `business` | Policy violation, insufficient funds, outside return window | Do not retry — explain the constraint to the user |
| `permission` | Unauthorized access, token expired, insufficient scope | Do not retry — escalate or inform the user |

The pattern is: `transient` errors are recoverable by waiting and trying again. Every other category requires changing something — the input, the request, or the user's situation — before a retry makes sense.

### Bad vs good error responses

The difference between a recoverable and unrecoverable error is often just the quality of the response metadata.

```json
// Bad — agent receives no actionable information
{
  "isError": true,
  "content": [{"type": "text", "text": "Operation failed"}]
}
```

With only "Operation failed," the agent cannot determine whether to retry, stop, or ask the user for help. It has to guess.

```json
// Good — every field serves the recovery decision
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "Order lookup failed: Database connection timeout after 5000ms",
    "errorCategory": "transient",
    "isRetryable": true,
    "retryAfterMs": 2000,
    "attemptedQuery": {"customer_id": "C-12345", "order_status": "pending"},
    "partialResults": null
  }]
}
```

Each field earns its place: `errorCategory` drives the retry decision, `retryAfterMs` tells the agent how long to wait, `attemptedQuery` lets it reconstruct the call without asking the user again, and `partialResults: null` confirms there is no usable partial data.

For a business rule violation, the metadata looks different because the recovery path is different — explain to the user, not retry:

```json
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "Refund denied: Order #98765 is outside the 30-day return window (order date: 2024-11-01).",
    "errorCategory": "business",
    "isRetryable": false,
    "customerMessage": "This order is no longer eligible for a refund as it was placed more than 30 days ago."
  }]
}
```

The `customerMessage` field gives the agent language it can use directly, rather than forcing it to reformulate the business rule from scratch.

## The critical distinction: access failure vs valid empty result

This is the most important conceptual distinction in MCP error handling, and it is frequently misunderstood.

An **access failure** means the tool could not do its job — the search service was down, the database was unreachable, the API returned a 500. The tool did not produce a result because something went wrong.

A **valid empty result** means the tool did its job and found nothing. The query ran successfully — there just were no matching records.

These must be represented differently:

```json
// Access FAILURE — something broke, agent should consider retry or fallback
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "Search service unavailable: connection refused on port 9200",
    "errorCategory": "transient",
    "isRetryable": true
  }]
}

// Valid EMPTY RESULT — the query succeeded, there is simply no data
{
  "isError": false,
  "content": [{
    "type": "text",
    "text": "No orders found for customer C-12345 with status 'pending'"
  }]
}
```

If you return `isError: false` when the search system was actually down — masking it as an empty result — the agent will tell the user "you have no pending orders" when the real answer is "we couldn't check." That is a correctness failure, not just a quality issue.

If you return `isError: true` when the query simply returned zero results, you trigger retry logic unnecessarily and confuse the agent about whether a real problem occurred.

### Subagent error propagation

In multi-agent systems, subagents should attempt local recovery for transient errors before escalating. Only propagate errors that the subagent genuinely cannot resolve. When propagating, include what was attempted and any partial results — this gives the orchestrating agent enough context to decide its next move without re-running completed work.

## Key things to know

- `isError: true` signals a tool failure — it does not terminate the agent loop, it informs the agent's next decision
- The four categories (`transient`, `validation`, `business`, `permission`) each imply a distinct recovery strategy
- Only `transient` errors should trigger automatic retries — all others require changing something before retrying
- An empty result and a failed lookup are fundamentally different and must not be conflated
- Metadata fields like `errorCategory`, `isRetryable`, `retryAfterMs`, and `partialResults` are what make errors actionable

## Exam tips

- The access failure vs empty result distinction appears in scenario questions. Look for cases where returning `isError: false` with an empty content array is wrong because the service was actually unavailable.
- Questions about "what should the agent do next" hinge entirely on `errorCategory`. Transient → retry. Validation/business/permission → stop and handle.
- A common distractor: returning a rich error message in the `text` field but omitting `errorCategory`. The message is human-readable but the agent still cannot make a programmatic recovery decision.
- "Silently returning empty results" is always an anti-pattern when it masks a real failure — this will appear as a wrong answer option.
- Subagent propagation questions test whether you know to include context (what was attempted, partial results) when escalating — not just re-throw the raw error.
