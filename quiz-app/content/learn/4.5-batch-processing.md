## What is the Message Batches API?

The Messages API processes requests synchronously — you send a request, Claude responds, you wait. For tasks where latency doesn't matter (nightly reports, weekly audits, bulk document processing), that's inefficient. You're paying for synchronous processing speeds when you don't need them.

The **Message Batches API** processes requests asynchronously at half the cost. You submit a batch of up to 10,000 requests, Claude processes them over the next few hours, and you retrieve the results when done. The trade-off: no guaranteed latency (up to 24 hours), and no multi-turn tool calling within a single request.

## How it works

```
┌──────────────┐    ┌──────────────────┐    ┌─────────────────┐
│  Submit      │ →  │  Processing      │ →  │  Retrieve       │
│  batch       │    │  (async, hours)  │    │  results        │
│  of requests │    │  ≤24h typically  │    │  by custom_id   │
└──────────────┘    └──────────────────┘    └─────────────────┘
```

Each request in the batch gets a `custom_id` you provide — this is how you correlate results back to your original data when the batch finishes:

```python
import anthropic

client = anthropic.Anthropic()

requests = []
for i, document in enumerate(documents):
    requests.append({
        "custom_id": f"doc_{document['id']}",   # Your correlation key
        "params": {
            "model": "claude-sonnet-4-6",
            "max_tokens": 1024,
            "tools": [extraction_tool],
            "tool_choice": {"type": "tool", "name": "extract_invoice_data"},
            "messages": [{"role": "user", "content": document["content"]}]
        }
    })

batch = client.beta.messages.batches.create(requests=requests)
batch_id = batch.id
```

Polling for completion and retrieving results:

```python
import time

while True:
    batch_status = client.beta.messages.batches.retrieve(batch_id)
    if batch_status.processing_status == "ended":
        break
    time.sleep(60)

# Process results — use custom_id to correlate
results = {}
failed_ids = []

for result in client.beta.messages.batches.results(batch_id):
    if result.result.type == "succeeded":
        results[result.custom_id] = result.result.message
    elif result.result.type == "errored":
        failed_ids.append(result.custom_id)
        if result.result.error.type == "context_too_long":
            # Resubmit with chunked document
            pass
```

## When to use batch vs synchronous

| Workflow | API | Reason |
|---|---|---|
| Developer waiting for PR review feedback | Synchronous | Sub-minute response needed |
| Nightly technical debt analysis across entire codebase | Batch | Latency-tolerant, large volume |
| Weekly security audit of all modified files | Batch | Non-blocking, runs overnight |
| Real-time customer support response | Synchronous | Customer waiting |
| Monthly invoice processing for thousands of documents | Batch | 50% cost saving, hours is fine |

The deciding question: *Can the person or system waiting tolerate hours?* If yes, batch. If no, synchronous.

## SLA calculation

If your SLA requires results within N hours and batch processing can take up to 24 hours, you must submit at most every **(N − 24)** hours.

Example: results needed within 30 hours → submit at least every 6 hours (30 − 24 = 6). Submitting more frequently than your window can fill risks missing the SLA.

## Batch testing strategy

Test on 10–20 documents before processing thousands. Fix prompt issues on the sample first — a batch resubmission across thousands of documents because of a prompt bug is expensive.

## Key things to know

- Message Batches API costs 50% less than synchronous at the cost of no latency guarantee (up to 24 hours)
- Each request needs a `custom_id` for result correlation — this is your responsibility
- Multi-turn tool calling is **not supported** within a single batch request
- Test on a sample before running bulk batches to avoid expensive resubmissions
- `processing_status == "ended"` means the batch is done — some individual requests may still have errors

## Exam tips

- "Need results in 5 minutes" → synchronous, not batch (batch has no latency guarantee)
- "50,000 invoices to process overnight" → batch (latency-tolerant, 50% cost saving)
- Multi-turn tool use required (tool call → tool result → follow-up) → cannot use batch API for this
- `custom_id` is for *your* correlation of results — not a Claude-side identifier
