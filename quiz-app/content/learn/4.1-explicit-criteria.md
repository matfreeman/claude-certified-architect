## What are explicit criteria?

When you ask Claude to do a review — code review, document review, data quality check — you're asking it to make judgment calls: is this a problem or not? Those judgment calls are only as reliable as the criteria you provide.

Vague instructions like "be conservative" or "only flag high-confidence issues" sound specific, but Claude's interpretation of "conservative" is subjective and not calibrated to your actual standards. The result is inconsistent behaviour: Claude flags things you don't care about, misses things you do, and the false positive rate erodes developer trust in the tool.

**Explicit criteria** replace subjective qualifiers with unambiguous categories and examples. Instead of "be conservative," you list exactly what to flag and exactly what to skip.

## How it works

Compare vague vs. specific instructions for code review:

```
BAD:  "Be conservative in your findings"
BAD:  "Only report high-confidence issues"
BAD:  "Focus on important issues"

WHY: Claude's interpretation of "conservative" is subjective —
     it's not calibrated to your codebase or your team's standards.
```

```
GOOD: "Report ONLY the following categories:
  - Security vulnerabilities (SQL injection, XSS, authentication bypass, exposed secrets)
  - Functional bugs (incorrect business logic, missing null checks that cause exceptions)
  - Data integrity issues (missing transactions, race conditions)

DO NOT REPORT:
  - Style issues or naming conventions
  - Patterns that differ from your preference but work correctly
  - Local patterns that are intentional (e.g., verbose error messages in this codebase)
  - Theoretical concerns without clear exploit paths"
```

## Explicit severity definitions

When you need Claude to classify findings by severity, give it concrete examples of each level:

```
CRITICAL: Code exploitable without authentication
  Example: SELECT * FROM users WHERE email = '{user_input}'  (SQL injection)

HIGH: Authentication/authorization bypass or data corruption
  Example: Missing admin check before deleting user records

MEDIUM: Logic errors causing incorrect results in specific scenarios
  Example: date.getMonth() without +1 (off-by-one in month display)

LOW: Minor issues with easy workarounds
  Example: Inefficient loop replaceable with a built-in function
```

With these definitions, Claude classifies consistently because it's matching patterns against examples, not guessing what "high" means.

## Managing false positive categories

If one category (e.g., "documentation gaps") has a high false positive rate, it damages trust in the entire review output — even findings in other categories that are accurate. The fix:

1. Identify which category is generating false positives
2. Temporarily disable that category while improving its criteria
3. Run samples of that category's output against ground-truth labels
4. Refine the definition until the false positive rate is acceptable
5. Re-enable

Never disable a whole review pipeline because one category is noisy. Isolate and fix the category.

## Key things to know

- Vague qualifiers ("conservative", "important", "high-confidence") are unreliable — Claude's interpretation is not calibrated to your context
- Explicit category lists (what to flag, what to skip) produce consistent, predictable behaviour
- Severity definitions need concrete examples for each level, not just labels
- High false positive rates in one category undermine trust in all categories
- The fix for false positives is always in the criteria, not in asking Claude to "try harder"

## Exam tips

- "Claude flags too many low-confidence issues" → add explicit DO NOT REPORT categories, not `tool_choice` changes
- Severity definitions without examples → unreliable classification (Claude will guess)
- A single noisy category → isolate and fix that category's criteria, don't disable the whole review
- The exam consistently frames the "lightest effective fix" as a prompt/criteria change, not an architecture change
