## What is MCP?

The Model Context Protocol (MCP) is an open standard that defines how external tools, data sources, and services connect to AI systems. Instead of every team writing their own custom integration layer, MCP provides a consistent protocol that Claude Code understands natively.

From a practical standpoint, an MCP server is a small process — usually Node.js or Python — that exposes tools and resources over a standardized interface. Claude Code discovers these servers at startup, connects to them, and makes their capabilities available in the same way as built-in tools. You can use a community-built GitHub server, a database server, and a custom internal server simultaneously, with no special integration code on the Claude side.

The configuration question is: where do you declare which servers to run?

## Two configuration locations

MCP servers are configured in one of two places, depending on who the configuration is for.

```
Project root/
├── .mcp.json          ← shared with the whole team via version control
└── src/...

~/.claude.json         ← personal to you, never committed
```

### Project-scoped: `.mcp.json`

Place this file in your project root and commit it. Every developer who checks out the project gets the same MCP server configuration automatically. This is the right choice for team tooling — GitHub integrations, internal databases, project-specific APIs.

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "database": {
      "command": "python",
      "args": ["-m", "mcp_server_db"],
      "env": {
        "DB_CONNECTION_STRING": "${DB_CONNECTION_STRING}"
      }
    }
  }
}
```

Notice the `${GITHUB_TOKEN}` syntax. This is environment variable expansion — the actual secret is never in the file. Each developer provides their own credentials via environment variables. The `.mcp.json` file describes the shape of the configuration; the secrets stay in each developer's local environment.

Never commit credentials directly into `.mcp.json`. The file goes into version control — anything you put there becomes part of the repository history.

### User-scoped: `~/.claude.json`

This file lives in your home directory and is personal to you. It is the right place for experimental servers you are building, personal productivity tools, or integrations that only make sense for your workflow.

```json
{
  "mcpServers": {
    "my-experimental-tool": {
      "command": "node",
      "args": ["/Users/me/dev/my-mcp-server/index.js"]
    }
  }
}
```

This server will be available in all your Claude Code sessions but will not appear for any other developer. When an experimental server graduates to a team tool, move its configuration to `.mcp.json`.

### At connection time

All configured MCP servers — from both locations — are discovered and connected when Claude Code starts a session. Their tools are available simultaneously alongside built-in tools. There is no runtime switching between server configurations.

## Tools vs Resources

MCP exposes two distinct concepts to Claude: **tools** and **resources**. Understanding the difference matters because they serve different purposes and behave differently in the protocol.

```
MCP Server
├── Tools     — functions Claude calls on demand, take action or query data
└── Resources — content catalogs Claude can inspect to understand what's available
```

**Tools** are executable. When Claude calls a tool, the server runs code and returns a result. Tools are how you get work done — fetching an issue, querying a database, sending a message.

**Resources** are declarative. They expose a catalog of available data — which issues exist, which files are indexed, which tables are queryable — without requiring Claude to make individual tool calls to discover them. Resources are identified by URIs and discovered separately from tools.

```
# Resource: expose what's available upfront
resource: issue_catalog
  URI: mcp://github/issues
  Returns: list of open issues with IDs, titles, and statuses
  Benefit: Claude sees the landscape without exploratory tool calls

# Tool: fetch specific data on demand
tool: get_issue_detail
  Input: issue_id
  Returns: full issue body, comments, assignees, labels
```

An important protocol nuance: resources are application-driven. The host — Claude Code or your application — decides how to surface, search, or pre-include them. The MCP protocol does not mandate a specific UX for resources; it just defines how they are identified and retrieved.

## Community servers vs custom servers

A large and growing registry of community MCP servers covers common integrations: GitHub, GitLab, Slack, Jira, Linear, PostgreSQL, and many more. For standard integrations, reach for a community server first.

Build a custom MCP server when:
- Your integration is specific to internal tooling that no community server covers
- You need to wrap proprietary APIs with custom logic or authentication
- You want to expose a constrained interface to sensitive data (returning summaries instead of raw records, for example)

When you do build a custom server, treat tool descriptions with the same care as any other tool definition. If your MCP tool has a weak description, Claude will prefer built-in alternatives — like `Grep` or `Read` — over your more capable server-side tool. A strong description is what makes the MCP tool win the selection decision.

## Key things to know

- `.mcp.json` is committed to version control and shared across the team — never put secrets in it
- `~/.claude.json` is personal and local — the right place for experimental or personal servers
- Use `${ENV_VAR}` syntax in `.mcp.json` to reference credentials without committing them
- All configured servers are available simultaneously — there is no per-session switching
- Resources expose available data upfront via URIs; tools execute on demand
- Weak MCP tool descriptions cause Claude to fall back to built-in tools — descriptions matter here too

## Exam tips

- The team vs personal distinction is the most common question pattern: `.mcp.json` for shared team tooling, `~/.claude.json` for personal servers.
- Environment variable expansion (`${GITHUB_TOKEN}`) is the correct answer when a question asks how to handle credentials in `.mcp.json`. Hardcoding secrets is always wrong.
- Resources vs tools: resources are passive catalogs (things Claude can see), tools are active functions (things Claude can call). They serve different purposes and are discovered differently.
- "All servers are available simultaneously" — questions sometimes try to trick you into thinking Claude switches between servers per session or per tool call.
- The "weak MCP description causes fallback to built-in tools" pattern may appear as a debugging scenario: Claude is using `Grep` instead of your custom search tool.
