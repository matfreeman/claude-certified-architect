## What this lesson is

This is the missing backbone for the course: a full, self-contained guide to the Model Context Protocol (MCP) built from the official MCP documentation, then translated into the architecture and tradeoff decisions the exam actually tests.

If you only remember one thing, remember this: **MCP is the protocol for context exchange, not a product-specific feature and not an orchestration framework by itself.** The exam will often give you a realistic Claude scenario and ask you to choose the lightest, most reliable way to connect Claude to external systems. MCP is the standard vocabulary underneath those decisions.

The official MCP versioning page currently lists `2025-11-25` as the current protocol version. You do not need to memorize every revision, but you do need to understand that version negotiation happens during initialization and that not every peer will support every feature.

---

## What MCP is, and what it is not

The official MCP docs define MCP as an open standard for connecting AI applications to external systems. It standardizes how context and actions are exposed so hosts like Claude Code or Claude Desktop can connect to many different systems without every team inventing its own bespoke adapter format.

Just as important is what MCP does **not** do:

- It does **not** dictate how your application uses an LLM
- It does **not** dictate your UI
- It does **not** replace prompt design, tool design, or orchestration
- It does **not** make security automatic

MCP gives you a protocol contract. You still decide what to expose, how safely to expose it, and how the host should use it.

## The mental model: host, client, server

This distinction is foundational and appears in both the official MCP docs and the exam material.

| Participant | What it is | Example |
|---|---|---|
| **Host** | The AI application the user interacts with | Claude Code, Claude Desktop, an IDE, a custom app |
| **Client** | The protocol-level component the host creates to talk to one server | One MCP client per connected MCP server |
| **Server** | A program that exposes capabilities over MCP | Filesystem server, GitHub server, internal database server |

The official architecture overview makes one subtle point very clear:

- the **host** manages the overall user experience
- the **client** manages a single connection to a single server
- the **server** exposes context and actions

That means a host can connect to multiple servers at once by creating multiple clients.

```text
Claude Code (host)
├── client A -> GitHub MCP server
├── client B -> Jira MCP server
└── client C -> Internal docs MCP server
```

**Exam trap:** a server is not “a tool.” A server may expose many tools, many resources, and many prompts.

## The two layers of MCP

The official MCP architecture splits the protocol into two layers:

| Layer | What it handles |
|---|---|
| **Data layer** | JSON-RPC 2.0 messages, lifecycle, capabilities, tools, resources, prompts, notifications |
| **Transport layer** | How messages move, connection setup, framing, and auth |

### Data layer

This is where most exam-relevant MCP reasoning happens. The data layer covers:

- initialization and capability negotiation
- requests and responses over JSON-RPC 2.0
- discovery methods like `tools/list` and `resources/list`
- execution methods like `tools/call`
- notifications when something changes
- utility features such as progress, cancellation, ping, logging, and argument completion

### Transport layer

The official docs describe two main transport mechanisms:

| Transport | Best fit | Key idea |
|---|---|---|
| **STDIO** | Local servers on the same machine | Direct process communication over stdin/stdout |
| **Streamable HTTP** | Remote servers | HTTP POST with optional streaming via Server-Sent Events |

From an exam perspective:

- `stdio` is usually the local, low-overhead choice
- Streamable HTTP is the remote, networked choice
- auth belongs at the transport layer, not in the tool definition itself

One current-doc nuance worth knowing: modern MCP guidance centers on **Streamable HTTP** rather than the older “HTTP plus SSE” mental model as if those were two separate top-level transports. Streamable HTTP may use SSE for streaming, but the transport choice you should think in is `stdio` vs Streamable HTTP.

## Lifecycle and capability negotiation

MCP is **stateful**. That matters.

The connection starts with an `initialize` exchange where client and server negotiate:

- protocol version
- supported capabilities
- optional features like list change notifications

The official docs emphasize that if the two sides cannot agree on a compatible protocol version or capability set, the connection should not continue.

That leads to a practical exam rule:

- when a feature is not working, check whether the capability was actually declared during initialization before assuming the tool or prompt is broken
- when two systems cannot agree on a protocol version, the session should not continue

Examples of capability declarations include:

- server declares `tools`, `resources`, or `prompts`
- client declares `roots`, `sampling`, or `elicitation`

**Debugging pattern:** if a server tries to use sampling or elicitation against a client that never declared those capabilities, the problem is capability negotiation, not prompt wording.

## The three server primitives

These are the core of MCP. The official server docs frame them as three different building blocks with three different controllers.

| Primitive | What it is | Who controls it |
|---|---|---|
| **Tools** | Executable actions | Model |
| **Resources** | Context data | Application |
| **Prompts** | Reusable prompt templates | User |

This control model is one of the cleanest ways to reason about MCP design decisions.

### Tools

Tools are executable functions. The model can discover them and invoke them automatically.

Official MCP guidance:

- tools are schema-defined
- inputs are validated with JSON Schema
- tools are for actions or queries that do work
- tools may require user approval depending on host behavior
- tool metadata can also carry annotations that help describe behavior such as whether a tool is read-only or destructive

Typical tool methods:

- `tools/list`
- `tools/call`

Examples:

- query a database
- create a calendar event
- post to Slack
- fetch issue details from Jira

**Exam connection:** tool descriptions are the primary signal the model uses to choose among tools. That is why Task Statements 2.1, 2.3, and 2.4 connect so tightly.

### Resources

Resources are not actions. They are structured, addressable context.

Official MCP guidance:

- resources are identified by URIs
- resources can be text or binary
- resources declare MIME types
- resources can be direct resources or parameterized templates
- resources are application-driven, not model-controlled

Typical resource methods:

- `resources/list`
- `resources/templates/list`
- `resources/read`
- `resources/subscribe`

Examples:

- `file:///repo/README.md`
- `calendar://events/2024`
- `schema://database/customers`
- `travel://flights/{origin}/{destination}`

The most exam-relevant idea here is this:

- use a **resource** when the host needs visibility into available context
- use a **tool** when the model needs to take an action or run an operation

That is why “content catalog” is such a strong exam phrase. If the issue tracker structure rarely changes and the agent keeps making exploratory calls just to learn what exists, a resource catalog is often the best architectural improvement.

### Prompts

Prompts are reusable templates exposed by the server, but they are **user-controlled** rather than automatically invoked by the model.

Official MCP guidance:

- prompts are discovered with `prompts/list`
- retrieved with `prompts/get`
- can take arguments
- can support parameter completion
- are often surfaced like slash commands, but the protocol does not require that specific UI

Prompts are useful for:

- standard workflows
- domain-specific guided tasks
- giving users structured entry points into tools and resources

**Exam connection:** prompts are not a substitute for deterministic enforcement. They guide workflows, but they do not create guarantees.

## Tools vs resources vs prompts: the exam version

Use this rule set under pressure:

- **Need to take action?** Use a tool.
- **Need to expose stable context or a catalog?** Use a resource.
- **Need a reusable, user-invoked workflow template?** Use a prompt.

And remember the ownership pattern:

- tools are **model-controlled**
- resources are **application-driven**
- prompts are **user-controlled**

That distinction is one of the fastest ways to eliminate wrong answers.

## Resource templates, subscriptions, and notifications

The official docs add a few details that are easy to miss but useful for architecture questions:

- resources may be fixed or template-driven
- resource templates help users discover valid parameter shapes
- dynamic resources can support parameter completion
- servers can support subscriptions so clients get notified when resource contents change
- lists of tools, prompts, and resources can also change dynamically via `listChanged` notifications

This matters because MCP is not just “static tool definitions at startup.” It supports dynamic discovery and real-time updates.

**Exam implication:** if the question is about reducing repeated exploratory work or keeping a host aware of changing available items, notifications and resources are often more correct than bolting everything into prompt instructions.

## The three client features learners usually miss

The official MCP docs also define client-side capabilities that many course overviews skip. These are worth knowing even when the exam focuses more heavily on server integration.

### Roots

Roots let the client tell a server which filesystem locations are in scope.

Official MCP guidance:

- roots are filesystem paths using `file://`
- servers can request them with `roots/list`
- clients can emit `roots/list_changed`
- roots are a **coordination mechanism**, not a hard security boundary

That last point matters a lot:

- roots help trusted servers stay focused
- roots do **not** replace OS permissions, sandboxing, or real access control

**Exam-ready phrasing:** roots are for scoping and accident prevention, not for hostile-server containment.

### Sampling

Sampling lets a server ask the client to perform an LLM completion on its behalf.

Official MCP guidance:

- this lets servers stay model-independent
- the client stays in control of model access and permissions
- sampling requests should keep a human in the loop
- users should be able to review or deny requests

This is especially useful when a server needs AI help but should not embed a model SDK or manage model credentials itself.

### Elicitation

Elicitation lets a server request structured user input through the client.

Official MCP guidance:

- requests are schema-driven
- clients should clearly show which server is asking and why
- users should be able to review, decline, or cancel
- servers **must not** use elicitation to request sensitive info

This enables interactive workflows without falling back to vague text prompts like “please ask the user for more details somehow.”

## Claude Code integration: where the exam gets practical

The official MCP site explains the protocol. The exam also expects you to know how Claude Code surfaces it in practice.

## Three practical scopes

Claude Code uses three scopes that map to two files:

| Scope | Shared? | Stored in | Best for |
|---|---|---|---|
| `local` | No | `~/.claude.json` under the current project | Private, current-project experiments |
| `project` | Yes | `.mcp.json` | Shared team tooling |
| `user` | No | `~/.claude.json` | Personal tooling across all projects |

For exam answers, the simplification is:

- shared team config -> `.mcp.json`
- private config -> `~/.claude.json`

### Environment variables

For shared `.mcp.json`, use environment variable expansion such as:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

Correct exam instincts:

- commit the config, not the secret
- use `${VAR}` or `${VAR:-default}` for expansion
- do not hardcode tokens

### Scope precedence

If the same server name exists in multiple scopes, the more specific scope wins:

1. local
2. project
3. user

### Discovery in Claude Code

All configured servers are available together. Claude does not “switch servers” one at a time.

That means a single session can use:

- built-in tools
- tools from multiple MCP servers
- resources from multiple MCP servers

The current Claude Code behavior also uses on-demand tool discovery, which means large MCP installations do not have to dump every tool definition into context immediately.

## Community server or custom server?

This comes up repeatedly in the study guide and is a classic exam tradeoff.

Prefer a community server when:

- the integration is standard
- the ecosystem already has a well-maintained solution
- you do not need company-specific workflow logic

Prefer a custom server when:

- you are wrapping internal systems
- you need constrained behavior for safety or compliance
- the workflow is unique enough that a generic server would be awkward or risky

**Exam pattern:** choose the existing community server for GitHub, Jira, Slack, or similar common tools unless the scenario explicitly requires internal business logic or specialized boundaries.

## Why agents ignore your MCP tool and use Grep instead

This is one of the most exam-relevant practical problems.

If Claude keeps choosing a built-in tool like `Grep` over a more capable MCP tool, the first fix is usually **not**:

- add a bigger model
- add a classifier
- add more examples everywhere
- build a wrapper server

The first fix is usually:

- improve the MCP tool name and description
- clarify inputs, outputs, examples, and boundaries
- explain why to use it instead of similar tools

This directly connects Task Statement 2.4 back to 2.1 and 2.3.

## The full exam-ready MCP design pattern

When you design an MCP integration well, it often looks like this:

1. Expose a **resource** for stable catalogs or schemas
2. Expose **tools** for actions or focused queries
3. Expose a **prompt** if users benefit from a guided template
4. Give agents only the tools they actually need
5. Return structured errors through `isError` when execution fails
6. Normalize ugly third-party outputs with hooks if the tools cannot be changed

That is the architecture the exam keeps steering you toward:

- explicit contracts
- least privilege
- stable context surfaces
- deterministic enforcement where risk is real

## Debugging and implementation basics from the official docs

The official MCP debugging guide recommends starting with **MCP Inspector**.

Inspector helps you:

- connect to stdio or Streamable HTTP servers
- inspect available tools, resources, and prompts
- invoke them directly
- watch notifications
- validate capability negotiation

That makes it the best first stop when a server “doesn’t work.”

### Logging gotcha for STDIO servers

The official build-server guide highlights a critical implementation detail:

- for `stdio` servers, never write logs to stdout
- stdout is reserved for JSON-RPC messages
- log to stderr instead

If you forget this, you can corrupt the protocol stream and break the server in ways that look mysterious from the client side.

### Good debugging workflow

The official debugging flow is:

1. test the server standalone with Inspector
2. check logs
3. verify initialization and capability exchange
4. then test in the target client

This is a much better workflow than trying to debug everything from inside the host UI first.

## Security and authorization

The official MCP docs are clear that authorization is **optional**, but strongly recommended for remote servers that expose sensitive resources or operations.

Use authorization when:

- the server exposes user-specific data
- the server performs admin or high-impact actions
- you need auditability or per-user access control
- the server is remote and multi-user

### Remote vs local auth expectations

The official docs distinguish the common cases clearly:

- **remote HTTP servers** should follow OAuth-style authorization flows
- **local stdio servers** typically use environment-based or locally acquired credentials instead

So on the exam:

- do not force remote OAuth logic into a local stdio question
- do not assume local roots or local config files are enough security for remote servers

### Security best-practice themes

The official security guidance calls out several risks that are useful to remember at a high level:

- **token passthrough is an anti-pattern**
- **scope minimization matters**
- **local servers are code execution on the user’s machine**
- **roots are not a security boundary**
- **users should understand what a server can do before connecting it**

For remote servers, verify authenticity before connecting. For local servers, treat installation and startup commands as privileged operations.

## The confusing “Task” terminology

One final nuance for exam prep:

- the official Anthropic exam materials may use `Task` to mean the subagent-spawning tool in the Agent SDK
- the official MCP architecture docs also mention experimental protocol-level **Tasks**

These are not the same concept.

If the scenario is about coordinator/subagent orchestration in Claude tooling, `Task` refers to the Anthropic subagent tool, not the experimental MCP protocol feature.

## Key things to know

- MCP standardizes context exchange between hosts and external systems; it does not prescribe your orchestration or UX.
- The host is the user-facing AI app, the client is the per-server connection component, and the server exposes capabilities.
- MCP has a data layer and a transport layer; lifecycle, capabilities, and primitives live in the data layer.
- The three server primitives are tools, resources, and prompts, and they have different controllers and different use cases.
- Resources are often the best answer when the problem is repeated discovery of stable data catalogs.
- Roots help scope filesystem work, but they are advisory rather than a hard security boundary.
- Sampling and elicitation are client capabilities that let servers ask for model completions or structured user input.
- In Claude Code, shared team MCP config belongs in `.mcp.json`; private config belongs in `~/.claude.json`.
- Environment variable expansion is the right answer for secrets in shared MCP config.
- Community servers are preferred for standard integrations; custom servers are for internal or specialized workflows.
- MCP Inspector is the best first debugging tool, and stdio servers must keep stdout clean.

## Exam tips

- If the scenario says “stable catalog,” “documentation hierarchy,” or “current sprint issues,” think **resource** before tool.
- If the scenario says “take action,” “mutate data,” or “run an operation,” think **tool**.
- If the question asks how to share MCP config across a team, the answer is usually project-scoped `.mcp.json` with environment variables.
- If Claude keeps choosing built-in tools over your MCP tools, improve the MCP tool descriptions before inventing more architecture.
- If a question mixes up host and client, remember: the host owns the UX, the client owns one connection.
- If a question suggests roots as a security control, that is a distractor. Roots scope behavior, but they do not enforce trust by themselves.
- If a question asks for the “best first debugging step,” choose Inspector and capability verification before rewriting prompts.
- If a scenario needs secure remote access to user data, authorization is likely required; for local stdio servers, environment-based credentials are more typical.
- If you see both “Task” and MCP in the same question, check whether it means Anthropic subagents or the experimental MCP task primitive.
