## What is tool design?

When you give Claude a set of tools, it has to read through them and decide which one to call — based entirely on their definitions. Claude doesn't run the code, test the functions, or peek at what the tools actually do. It reads the name, description, and schema, and makes a judgment call.

That means the quality of your tool definitions directly determines how reliably Claude picks the right tool. A well-designed tool definition is essentially a contract between you and Claude: here is what this tool does, when to use it, and what it expects. Write it poorly and Claude will guess, often wrong.

This is one of the most leverage-rich areas in agentic system design. The same underlying functions — identical code — can behave dramatically differently depending on how their tool definitions are written.

## How it works

When Claude encounters a tool call decision, it processes the candidate tools much like it processes any other text: looking for meaning, context, and differentiation. Two tools that "analyze documents" look identical to Claude. Two tools that "extract structured data from web pages" vs. "parse academic PDFs into sections and citations" are clearly distinct.

### The anatomy of a good tool description

A description that enables reliable selection has four components:

1. **What it does** — the core action in plain language
2. **When to use it** — the specific trigger conditions
3. **Input format** — what the tool actually expects
4. **Negative guidance** — what NOT to use it for, with redirects

Here is the difference in practice:

```python
# Bad — Claude cannot distinguish these tools
tools = [
    {"name": "analyze_content", "description": "Analyzes content"},
    {"name": "analyze_document", "description": "Analyzes documents"},
]
# Result: Claude will misroute frequently. The names are different but the
# descriptions give zero information for differentiation.
```

```python
# Good — each tool is unambiguous
tools = [
    {
        "name": "extract_web_results",
        "description": """Extracts structured data from web search result pages.

        Use when: Processing HTML or plain text from web URLs, news articles, blog posts.
        Input format: Raw HTML or plain text fetched from a web page.
        Output: Structured JSON with title, author, date, key claims, source URL.

        Do NOT use for: PDF documents, database records, or API responses.
        Instead use: extract_document_data for PDFs, query_database for structured records."""
    },
    {
        "name": "extract_document_data",
        "description": """Extracts structured data from documents (PDF, Word, research papers).

        Use when: Processing academic papers, reports, contracts, or whitepapers.
        Input format: Document content as text or base64-encoded binary.
        Output: Structured JSON with sections, citations, methodology, and findings.

        Do NOT use for: Web pages or raw HTML content.
        Instead use: extract_web_results for anything fetched from a URL."""
    }
]
```

The negative guidance ("Do NOT use for") is not just defensive — it actively helps Claude navigate a decision tree. When Claude is uncertain, these redirects resolve the ambiguity.

### When to split tools

A common mistake is building one generic tool and hoping the description handles all the cases. Generic tools create selection ambiguity and often lead to over-use of a single tool for tasks it handles poorly.

Splitting a generic `analyze_document` into purpose-specific tools produces more reliable behavior:

```python
tools = [
    {
        "name": "extract_data_points",
        "description": "Extracts quantitative data points, statistics, and measurements from text."
    },
    {
        "name": "summarize_content",
        "description": "Creates a concise summary of the main argument or narrative of a document."
    },
    {
        "name": "verify_claim_against_source",
        "description": "Checks whether a specific factual claim is supported, contradicted, or unaddressed by source text."
    },
]
```

The right signal for splitting: if you find yourself writing "use this tool when X, but also when Y, and sometimes Z" — that is a single tool trying to do multiple jobs. Split it.

### The `input_examples` field

For tools with complex inputs — nested objects, optional parameters with non-obvious formats, or enums where the right choice isn't clear — you can provide `input_examples` alongside the schema. These give Claude concrete patterns to follow when constructing the tool call.

```json
{
  "name": "get_weather",
  "description": "Get current weather for a location. Use for live weather lookups only, not forecasts.",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {"type": "string"},
      "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
    },
    "required": ["location"]
  },
  "input_examples": [
    {"location": "San Francisco, CA", "unit": "fahrenheit"},
    {"location": "Melbourne, VIC", "unit": "celsius"}
  ]
}
```

`input_examples` are a supplement, not a replacement. They clarify format and valid values — they do not substitute for a clear description of what the tool does and when to call it.

### System prompt keyword interference

This one is subtle and catches many developers off-guard. The wording of your system prompt can create accidental associations that override even well-written tool descriptions.

Consider a system prompt that says: "Always check customer records before processing any orders." Now when a user asks about an order, Claude reads "customer" in the system prompt, sees a tool named `get_customer`, and may route order queries to that tool — even though you have a `lookup_order` tool that is the correct choice.

The system prompt created an implicit link between the word "customer" and the `get_customer` tool. Claude follows that link.

The fix is to audit your system prompt for keywords that appear in tool names, and either rename the tools to eliminate the overlap or rewrite the system prompt to use neutral language:

```
# Problematic system prompt
"Always check customer records before processing any orders."

# Rewritten to avoid keyword collision
"Always retrieve the relevant account data before processing requests."
```

## Key things to know

- Descriptions are the primary mechanism Claude uses to select tools — not names, not schemas
- Negative guidance ("do NOT use for X, use Y instead") is a reliable way to resolve ambiguity between similar tools
- `input_examples` help Claude format inputs correctly for complex schemas — they belong alongside a good description, not instead of one
- The right time to split a tool is when its description needs conditional logic ("use when X, but not Y")
- System prompt keywords can create accidental tool associations — audit both the prompt and tool names together

---
## Exam tips

- Questions often present a pair of tool descriptions and ask why one causes misrouting. Look for descriptions that lack specificity, missing negative guidance, or keyword overlap with the system prompt.
- `input_examples` is a real field in the Anthropic API tool definition — don't confuse it with the `description` or the JSON schema's `examples` property.
- "Split the tool" is often the correct answer when a description has to juggle multiple use cases — but the distractor will suggest improving the description instead.
- System prompt interference is a less obvious failure mode than bad descriptions. In scenario questions, check whether the system prompt contains words that match tool names.
- Tool name clarity matters, but a clear name alone won't fix a missing description. Both matter.
