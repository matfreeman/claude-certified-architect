## What is few-shot prompting?

Few-shot prompting means including worked examples directly in your prompt — showing Claude the *input* and the *expected output* before asking it to handle a new input. Instead of describing the rules, you demonstrate them.

This is one of the most reliable techniques for achieving consistent output, especially for tasks where:
- Detailed instructions produce inconsistent formatting
- The task involves ambiguous cases that need concrete precedent
- You're extracting structured data and want to reduce hallucination
- Claude needs to distinguish acceptable patterns from genuine problems

## How it works

The pattern is simple: embed examples in your system prompt in a clear format, then send the real task as the user message.

**Example: code review output format**

```python
system_prompt = """You are a code reviewer. Format every finding exactly as shown:

EXAMPLE 1:
File: src/auth/login.py, Line 47
Issue: SQL injection — user input directly concatenated into SQL query
Severity: CRITICAL
Suggested fix: Use parameterized queries: cursor.execute("SELECT * FROM users WHERE email = %s", (email,))

EXAMPLE 2:
File: src/api/orders.py, Line 123
Issue: Missing null check — order.customer can be None if order was created before customer verification
Severity: HIGH
Suggested fix: Add guard: if order.customer is None: return error_response("Customer not found")

EXAMPLE 3 (acceptable pattern — do NOT flag):
File: src/utils/dates.py, Line 34
Pattern: Explicit timezone conversion on every datetime operation
Note: This is intentional — the codebase deliberately avoids implicit timezone handling
Action: Skip — documented intentional pattern, not a bug

Now review the provided code using this exact format."""
```

The third example is critical — it teaches Claude what *not* to flag. Without it, Claude might report every timezone conversion as a potential issue.

## Resolving tool selection ambiguity

Few-shot examples work for decisions, not just output format. When Claude needs to choose between tools:

```python
system_prompt = """When selecting between lookup_order and get_customer, follow these examples:

User: "Check order #12345"
→ Call: lookup_order(order_id="12345")
Reasoning: User referenced an order ID — lookup_order is for order data.

User: "What's the status of my account?"
→ Call: get_customer(identifier=<from context>)
Reasoning: User asked about their account — get_customer is for account/profile data.

User: "Did my recent order ship?"
→ Call: get_customer(identifier=<from context>) THEN lookup_order(customer_id=<result>)
Reasoning: No order ID — must identify the customer first, then find their recent order."""
```

## Handling varied document structures

For extraction tasks where the source documents vary significantly, examples for each document type prevent the model from applying one document's structure to another:

```python
system_prompt = """Extract author and date from documents:

EXAMPLE 1 (academic paper):
Document: "...Published by Jane Smith (j.smith@university.edu) in March 2025..."
Result: {"author": "Jane Smith", "date": "2025-03"}

EXAMPLE 2 (blog post):
Document: "Posted on January 15, 2025 | By @janesmith"
Result: {"author": "janesmith", "date": "2025-01-15"}

EXAMPLE 3 (report with no author):
Document: "Q4 2024 Annual Report — Finance Division"
Result: {"author": null, "date": "2024-Q4"}

EXAMPLE 4 (imprecise measurement — handle gracefully):
Document: "The component weighs about 2-3 kilos"
Result: {"weight_kg": null, "weight_note": "approximately 2-3 kg, not exact"}"""
```

Example 4 is important — it sets a precedent for how to handle ambiguous input, preventing hallucinated precision.

## How many examples?

**Target 2–4 examples.** Focus them on ambiguous cases, not obvious ones.

| Count | Trade-off |
|---|---|
| 1 | Minimal disambiguation — Claude may still guess on edge cases |
| 2–4 | Strong signal, low token overhead — the sweet spot |
| 6–8 | Marginal improvement for significant token cost |
| 10+ | Rarely worth it unless document types are highly varied |

## Key things to know

- Few-shot examples are the most effective technique for achieving *consistently formatted* output
- Include examples of what *not* to flag — not just what to flag
- 2–4 examples is the sweet spot; more than 6–8 adds token cost without proportional benefit
- Focus examples on edge cases, not obvious cases — Claude handles the obvious ones without help
- Few-shot works for decisions (tool selection) and output format, not just extraction

## Exam tips

- "Claude formats output inconsistently despite detailed instructions" → add few-shot examples, not more prose
- "Claude flags intentional patterns as bugs" → add a NOT-flag example to the few-shot set
- Ambiguous tool selection → few-shot examples showing the decision, not more description in tool definitions
- The exam may ask about example count: the correct answer is 2–4, with focus on ambiguous cases
