## What is multi-instance review?

When you ask Claude to review something it just created — code, a document, an analysis — it still has the reasoning context from generating that output. It "remembers" why it made each decision, and that memory biases it toward defending those decisions rather than questioning them.

This is not a capability limitation or a bug. It's a fundamental property of how context works: any agent reviewing its own recent output will tend to agree with it. The fix is structural — use an independent review instance with no generation context.

## How it works

```
# WRONG — reviewer has generation context
code = generate_code(session, requirements)
review = review_code(session, code)    ← same session, biased

# CORRECT — fresh reviewer with no prior context
code  = generate_code(session_A, requirements)
review = review_code(session_B, code)  ← new session, independent
```

Session A never sees Session B's conversation, and vice versa. Session B approaches the code as if seeing it for the first time — because it is.

## Multi-pass review for large changesets

When reviewing a large PR (14+ files), a single-pass review produces:
- Inconsistent depth (detailed for some files, superficial for others)
- Attention dilution (middle files get less attention — "lost in the middle" effect)
- Contradictions (flags a pattern as bad in file 3, approves the same pattern in file 11)

Split into passes:

```python
async def review_pr(files: list[str]) -> dict:
    # Pass 1: Analyze each file independently for local issues
    local_findings = {}
    for file in files:
        findings = await analyze_file_locally(file)
        local_findings[file] = findings
    
    # Pass 2: Cross-file integration pass
    integration_findings = await analyze_cross_file_issues(
        files=files,
        local_findings=local_findings,
        focus="data flow between modules, interface contracts, shared state"
    )
    
    return merge_findings(local_findings, integration_findings)
```

Pass 1 gives each file focused attention. Pass 2 looks at the files *together*, checking for issues that only appear at integration boundaries.

## Confidence-scored routing

Rather than accepting all findings or discarding all findings, route them by confidence:

```python
schema = {
    "findings": [{
        "issue": "string",
        "severity": "string",
        "confidence": {"type": "string", "enum": ["high", "medium", "low"]},
        "reasoning": "string"   # Why this confidence level
    }]
}

# Route based on confidence:
# high   → auto-comment on PR (no human needed)
# medium → human review queue
# low    → discard or separate review pipeline
```

This gives you a tunable threshold: raise it to reduce noise, lower it to catch more potential issues.

## Key things to know

- Self-review in the same session is biased by generation context — this is a structural issue, not fixable with prompting
- Independent review = new API session with no access to the generation conversation
- Multi-pass review: local pass (per-file) + integration pass (cross-file) beats a single full-scan
- Confidence-scored routing lets you separate auto-actionable findings from human-review candidates
- The "lost in the middle" effect applies to large PRs — attention dilutes across many files in one pass

## Exam tips

- "Claude approved the same bug it introduced" → independent review session (different session), not a better system prompt
- "Inconsistent review depth across a 20-file PR" → multi-pass review (local + integration), not a longer prompt
- Self-review limitation is about *context retention*, not model capability — prompting it to "review critically" doesn't fix it
- Confidence routing routes by the model's stated confidence per-finding, not by severity — the two are different dimensions
