## What is tool_choice?

When you give Claude a set of tools, it normally decides on its own whether to call one, which one to call, and when to respond with text instead. `tool_choice` is the parameter that lets you override this default behavior. It ranges from "you must call a tool" to "you may not call any tools at all," with stops in between.

Understanding `tool_choice` is half the story. The other half is a design principle: the tools you give Claude — and how many of them — directly affects how reliably it uses them. These two concerns fit together into a unified approach to tool distribution.

## The tool_choice options

| Value | What it means | When to use it |
|---|---|---|
| `"auto"` (default) | Claude decides whether to call a tool or respond with text | Normal usage — Claude picks the right moment |
| `"any"` | Claude must call a tool — any tool from the available set | When you need guaranteed structured output rather than prose |
| `{"type": "tool", "name": "..."}` | Claude must call this specific named tool | When you need a specific operation to happen before anything else |
| `"none"` | Claude may not call tools, even if they are present | When you want Claude to reason about tools without using them |

The most important distinction is between `"auto"` and the two forcing modes (`"any"` and the named tool object).

With `"auto"`, Claude may respond with text and never call a tool — that is a valid outcome. If your pipeline needs structured output or a specific action to happen, `"auto"` does not guarantee it.

With `"any"` or a named tool, Claude's response is prefilled to begin with a tool call. There will be no natural-language preamble before the `tool_use` block. If your parsing logic expects that structure, forcing tool use removes ambiguity.

### Forced tool use and extended thinking

One important constraint: forcing tool use (`"any"` or a named tool object) is incompatible with extended thinking. If you are using extended thinking in your request, you must use `"auto"` or `"none"` for `tool_choice`.

### Guaranteeing execution order

Forcing a specific tool is particularly useful when you have a pipeline where one tool must produce output that feeds into another. You cannot rely on `"auto"` to ensure the first step happens:

```python
# Step 1: Force metadata extraction before anything else
response1 = client.messages.create(
    model="claude-sonnet-4-6",
    tools=[extract_metadata_tool, enrich_data_tool],
    tool_choice={"type": "tool", "name": "extract_metadata"},
    messages=[{"role": "user", "content": document_content}]
)

# Step 2: Now enrich — the metadata is available as prior context
response2 = client.messages.create(
    model="claude-sonnet-4-6",
    tools=[enrich_data_tool],
    tool_choice="auto",
    messages=[
        {"role": "user", "content": document_content},
        {"role": "assistant", "content": response1.content},
        {"role": "user", "content": [{"type": "tool_result", ...}]}
    ]
)
```

Step 1 forces `extract_metadata` to run. Step 2 can then be `"auto"` because the enrichment decision can be left to Claude — the mandatory first step has already been guaranteed.

### Combining strict schemas with forced tool use

When you need both a guaranteed tool call and schema-conformant inputs, the strongest pattern is `strict: true` on the tool definition combined with `tool_choice: "any"`. `strict: true` means Claude will always produce outputs that conform exactly to your JSON schema. Together, they give you predictable, parseable output every time.

## The principle of least privilege for tools

Giving an agent access to all available tools is tempting — it seems more capable. In practice, it is less reliable. The more tools Claude has to consider, the more likely it is to pick the wrong one, use a tool outside its intended scope, or produce unpredictable behavior at the boundaries between tools.

The principle of least privilege says: give each agent only the tools it actually needs for its specific role.

```python
agents = {
    "web-searcher": AgentDefinition(
        tools=["WebSearch", "WebFetch"],
        # Explicitly excluded: Read, Edit, Bash, Grep
        # A searcher does not need to modify files
    ),
    "doc-analyzer": AgentDefinition(
        tools=["Read", "Grep", "Glob"],
        # Explicitly excluded: WebSearch, Bash
        # An analyzer works on local files, not the web
    ),
    "synthesizer": AgentDefinition(
        tools=["Read", "verify_fact"],
        # verify_fact handles 85% of fact checks internally
        # Complex verifications route through the coordinator to web-searcher
    ),
}
```

The synthesizer example illustrates an important nuance: you can give a subagent a scoped cross-role tool for high-frequency operations while still routing complex cases through the proper channel. This eliminates the majority of round-trip overhead without fully collapsing the separation of concerns.

### Constrained tools vs generic ones

Another dimension of least privilege is replacing wide-open tools with constrained variants that enforce boundaries internally:

```python
# Generic tool — can fetch any URL, including arbitrary external destinations
tools = [
    {
        "name": "fetch_url",
        "description": "Fetches content from a URL."
    }
]

# Constrained variant — validates the URL before fetching
tools = [
    {
        "name": "load_document",
        "description": "Loads a document from validated document storage only. "
                       "Validates the URL against approved storage domains before fetching. "
                       "Will reject requests for arbitrary external URLs."
    }
]
```

The constrained tool does not rely on Claude making good decisions about which URLs are acceptable — the validation is built into the tool itself. This is safer and more predictable.

## Key things to know

- `"auto"` may return text instead of a tool call — use `"any"` or a named tool when you need guaranteed tool execution
- Forcing tool use is incompatible with extended thinking — use `"auto"` or `"none"` in that context
- `strict: true` plus `tool_choice: "any"` is the strongest pattern for guaranteed, schema-conformant output
- More tools does not mean more capable — it means more decision complexity and less reliability
- Constrained tool variants enforce boundaries inside the tool, not by relying on Claude's judgment

## Exam tips

- `"any"` forces a tool call but Claude picks which tool. A named tool object forces a specific tool. Know the difference.
- The extended thinking incompatibility with forced tool use is a common distractor — forced tool use requires `"auto"` or `"none"`, not the other way around.
- "Give the agent all tools to maximize flexibility" is a trap answer. More tools increases misuse and decision complexity.
- When a scenario describes a pipeline where step 2 depends on step 1, forcing a specific tool with `{"type": "tool", "name": "..."}` is the correct approach — not `"any"`.
- The synthesizer with a scoped `verify_fact` tool tests whether you understand that least-privilege doesn't mean zero cross-role access — it means minimal, justified access.
