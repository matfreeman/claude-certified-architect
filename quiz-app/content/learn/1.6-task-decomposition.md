### 1.6 Task Decomposition Strategies

## What do we mean by "task decomposition"?

You're right not to expect a single Claude docs page with this exact title.

The exact phrase **"task decomposition strategies"** is not a standalone Claude docs page. But one of the core patterns inside this topic, **prompt chaining**, *is* documented directly in Anthropic's prompt-engineering guidance.

In this course, **task decomposition strategies** is an **exam-prep umbrella term** for one design decision:

- how you break a large problem into smaller units
- in what order those units should run
- which parts should stay in the main agent versus go to subagents
- when the next step should be chosen up front versus discovered dynamically

The official docs discuss the pieces separately:

- the **prompting best practices** docs explicitly cover **"Chain complex prompts"**
- the **subagents** docs cover context isolation, delegation, and parallel work
- **best practices** covers explore first, then plan, then implement
- the **context window** docs explain why large, noisy tasks need tighter scoping

Task Statement 1.6 pulls those pieces together into an architecture choice.

## Decomposition is not the same as execution

It helps to separate two questions:

| Question | Meaning |
|---|---|
| **How do we split the work?** | Decomposition |
| **How do we run the pieces?** | Execution topology: sequential, parallel, or mixed |

You might decompose a problem into four chunks, then:

- run them sequentially as a fixed chain
- run independent chunks in parallel with subagents
- re-plan after each result and create the next chunk adaptively

## Two exam patterns

| Pattern | When to use | What it looks like |
|---|---|---|
| **Prompt chaining** (fixed sequential) | Predictable, repeatable, multi-pass tasks | You know the steps up front and run them in a defined order |
| **Adaptive decomposition** | Open-ended investigations where each finding changes the plan | You discover structure first, then decide the next subtasks |

### Pattern 1: Prompt chaining

Use prompt chaining when the task is stable enough that the review passes are obvious before you start.

Example: a large code review.

```
Step 1: Analyze auth.py → findings_auth
Step 2: Analyze orders.py → findings_orders
Step 3: Analyze payments.py → findings_payments
Step 4: Cross-file integration pass → integrated_findings
Step 5: Prioritized report
```

This works well because each pass has a clear purpose. It also reduces **attention dilution**: if Claude reviews too many files and concerns at once, depth becomes uneven and cross-file reasoning gets muddy.

This pattern connects to the exam guide's example of per-file local analysis followed by a separate cross-file integration pass.

### Pattern 2: Adaptive decomposition

Use adaptive decomposition when you cannot know the best plan until you've explored the problem.

Example: "Add comprehensive tests to this legacy codebase."

```python
# Step 1: Map the codebase structure
# Step 2: Identify risky or undertested modules
# Step 3: Prioritize by impact, dependencies, and testability
# Step 4: Generate tests for the highest-value targets first
# Step 5: Re-plan as new coupling and edge cases are discovered
```

This matches the exam guide's wording: first map structure, then identify high-impact areas, then create a plan that adapts as dependencies emerge.

## How this maps to subagents

Subagents are one way to **execute** a decomposition, not the decomposition itself.

For example:

- the coordinator may decompose a research problem into source collection, document analysis, and synthesis
- once that scope is clear, the independent pieces can run in parallel via subagents
- if coverage gaps remain, the coordinator can adapt the plan and spawn new targeted subtasks

That is why Task Statement 1.6 overlaps with 1.2 and 1.3. The exam is testing judgment about how to break up the work, not recall of a single API.

## Key things to know

- "Task decomposition strategies" is an exam-study framing, not a named SDK feature
- Prompt chaining is directly documented by Anthropic, even though the broader exam phrase is not
- Prompt chaining is best for predictable multi-pass work
- Adaptive decomposition is best for open-ended investigation
- Decomposition decides the chunks; subagents are one way to execute those chunks
- Good decomposition reduces context bloat and attention dilution
- Bad decomposition often shows up as missing coverage, weak prioritization, or wasted parallel work

## Exam tips

- If the scenario is predictable and repeatable, fixed sequential passes are usually best
- If the scenario is exploratory, start by mapping the problem before committing to a full plan
- If a multi-agent system misses major parts of a topic, suspect bad decomposition by the coordinator
