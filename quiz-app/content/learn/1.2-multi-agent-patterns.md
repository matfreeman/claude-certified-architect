## What is the coordinator-subagent pattern?

As tasks grow more complex, a single agent running in a single loop hits limits. The context window fills up. Attention spreads thin across too many concerns. Some subtasks could run in parallel but instead run sequentially.

The **coordinator-subagent pattern** solves this by splitting the work. One agent — the coordinator — is responsible for understanding the overall task, breaking it down, and directing specialized subagents to handle each piece. The subagents do focused work and report back. The coordinator assembles the pieces into a final result.

This is sometimes called **hub-and-spoke**: the coordinator is the hub, and subagents are the spokes. All communication flows through the hub. Spokes never talk to each other directly.

Think of it like a project manager working with a team of specialists. The manager doesn't do the technical work — they assign it, collect deliverables, and synthesize a report. Each specialist works independently on their piece, with a clear brief from the manager.

```
                    │ COORDINATOR │

    ┌────────────┐  ┌────────────┐  ┌────────────┐

    │  Subagent  │  │  Analyzer  │  │  Subagent  │
    │  (spoke)   │  │  (spoke)   │  │  (spoke)   │
    └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
          │               │               │
          └───────────────┴───────────────┘

                    ┌─────────────┐
                    │ COORDINATOR │
                    │ aggregates  │
                    └─────────────┘
```

Notice that the subagents never communicate with each other. Web Search and Document Analyzer both return to the coordinator, which then passes combined findings to the Synthesizer. Everything passes through the hub.

### A concrete execution flow

Suppose the coordinator receives: "Research the impact of AI on creative industries."

1. **Decomposition:** The coordinator identifies three workstreams — web research on recent news, analysis of uploaded documents, and synthesis of all findings.
2. **Parallel dispatch:** The coordinator spawns the Web Search subagent and Document Analyzer subagent simultaneously, each with a focused prompt.
3. **Collection:** Both subagents complete and return structured results to the coordinator.
4. **Handoff:** The coordinator passes *all* collected findings explicitly in a new prompt to the Synthesizer subagent.
5. **Evaluation:** The coordinator reviews the synthesis, decides if it's sufficient, and either accepts it or re-delegates targeted follow-up queries.
6. **Delivery:** The coordinator returns the final answer to the user.

## Context isolation — the most important thing to understand

Subagents are **fully isolated**. They do not inherit the coordinator's conversation history, prior tool results, reasoning steps, or memory. Each subagent starts fresh with only what you put in its prompt.

This is not a limitation — it's by design. Isolation keeps subagent context windows small and focused. But it means **the coordinator is responsible for explicitly passing everything a subagent needs.**

If the coordinator has web search results that the synthesizer needs, the coordinator must include those results in the synthesizer's prompt. The synthesizer cannot reach back and read the coordinator's history.

```python
# WRONG — synthesizer cannot see web_results from parent context
subagent_prompt = "Synthesize the research findings"

# CORRECT — pass everything explicitly
subagent_prompt = f"""
Synthesize these research findings into a comprehensive report:

## Web Search Results
{json.dumps(web_results, indent=2)}

## Document Analysis
{json.dumps(doc_analysis, indent=2)}

Requirements:
- Preserve source URLs for every claim
- Distinguish well-established from contested findings
"""
```

## Coordinator responsibilities

A well-designed coordinator does six things:

**1. Task decomposition** — Breaking the user's request into focused, non-overlapping subtasks. This is harder than it sounds (see below).

**2. Dynamic subagent selection** — Choosing *which* subagents to invoke based on the specific query. Not every task needs all subagents. A question about recent news doesn't need the document analyzer if no documents are available.

**3. Scope partitioning** — Assigning distinct work to each agent to minimize duplication. If two subagents both search the web for the same terms, you get redundant results and wasted cost.

**4. Result aggregation** — Collecting outputs and combining them into a coherent picture. This often involves de-duplication, conflict resolution, and filling gaps.

**5. Iterative refinement** — Evaluating the synthesized result, identifying gaps, and re-delegating targeted follow-up queries until the coverage is sufficient. Good coordinators don't just accept the first synthesis.

**6. Error handling** — If a subagent fails or returns low-quality output, the coordinator decides how to recover: retry, use an alternative subagent, or degrade gracefully.

## The decomposition trap

Task decomposition seems straightforward, but getting it wrong poisons the entire system — even if every subagent executes perfectly.

Consider this scenario: a coordinator decomposes "research AI's impact on creative industries" into three subtasks:
- "AI in digital art"
- "AI in graphic design"
- "AI in photography"

Each subagent runs flawlessly. The synthesis is thorough and well-sourced. But the final report is still incomplete — music, writing, film, and architecture are entirely missing. The subagents did exactly what they were asked; the coordinator asked for the wrong things.

**The root cause of poor coverage is always the coordinator, not the subagents.** When a multi-agent system produces incomplete results, check the decomposition strategy before blaming execution quality.

Good decomposition is exhaustive within the scope it covers. For broad, open-ended topics, the coordinator may need to explicitly enumerate categories rather than rely on broad labels.

## Key things to know

- All communication flows through the coordinator — subagents never talk to each other
- Subagents have fully isolated context — they only know what their prompt tells them
- The coordinator must explicitly pass all required context in each subagent's prompt
- Parallel spawning happens by emitting multiple `Agent` tool calls in a single coordinator response
- Incomplete results from a multi-agent system usually mean flawed decomposition by the coordinator
- The coordinator is responsible for iterative refinement — evaluating outputs and re-delegating if needed
- Scope partitioning (assigning distinct work) prevents redundant effort across subagents

## Exam tips

- When a scenario describes incomplete results from a multi-agent system, look for the coordinator's decomposition as the root cause — even if subagents executed correctly.
- Questions may ask which agent is responsible when gaps appear in coverage. The answer is almost always the coordinator.
- Watch for answers that suggest subagents "inherit" context from the coordinator — they don't. Context must be passed explicitly.
- The hub-and-spoke topology means no subagent-to-subagent communication — all routing goes through the coordinator.
- "Dynamic subagent selection" means the coordinator chooses which agents to invoke per query, not that it always runs all of them.
