## What is workflow enforcement?

Agents are powerful, but they're also probabilistic. Claude follows instructions well — most of the time. When you say "always verify customer identity before processing a refund," Claude will comply in the vast majority of cases. But "vast majority" is not the same as "always."

For many tasks, that's fine. If Claude occasionally skips a formatting preference or writes a comment in a slightly different style than requested, nothing breaks.

But some workflows have real consequences. Processing a refund for an unverified customer could mean financial fraud. Deploying code to production before tests pass could break a live system. Sending a medical recommendation without checking drug interactions could cause harm. In these cases, "usually follows the rule" is not acceptable. You need **guaranteed enforcement**.

**Workflow enforcement** is the practice of using code — not prompts — to guarantee that required steps happen in the right order, with the right preconditions, every single time.

## How it works

The distinction comes down to where the rule lives:

```
PROMPT-BASED (probabilistic)
─────────────────────────────
System prompt: "Always call get_customer before process_refund"
                           │
                           ▼
              Claude reads this, understands it,
              follows it ~90-95% of the time.
              The other ~5-10%: the refund goes through.


PROGRAMMATIC (deterministic)
─────────────────────────────
PreToolUse hook on "process_refund":
  if not context["customer_verified"]:
      deny("Must verify customer first")
                           │
                           ▼
              process_refund cannot execute
              unless get_customer ran first.
              No exceptions. No probability.
```

The hook intercepts the tool call before it executes. If preconditions aren't met, the tool is blocked and Claude receives an explanation. Claude then has to take a different action — like calling `get_customer` first — before it can proceed.

### A prerequisite gate in practice

Here's what a verification gate looks like as a `PreToolUse` hook:

```python
async def enforce_verification_gate(input_data, tool_use_id, context):
    """Block refund processing until customer verification is complete."""
    if input_data["tool_name"] == "process_refund":
        if not context.get("customer_verified"):
            return {
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "permissionDecision": "deny",
                    "permissionDecisionReason": (
                        "Must call get_customer first to verify customer identity"
                    )
                }
            }

    return {}
```

When `process_refund` is attempted without verification, Claude receives a denial with a reason. Claude understands the reason and (correctly) calls `get_customer` first. A `PostToolUse` hook on `get_customer` then sets `context["customer_verified"] = True`, allowing `process_refund` to proceed on the next attempt.

The enforcement is in the code, not in Claude's judgment.

## When to use programmatic enforcement

Use hooks and gates when:

- **Financial consequences** — refunds, charges, transfers that can't be reversed without cost
- **Safety consequences** — medical, physical, or security-critical operations
- **Compliance requirements** — audit trails or regulatory steps that must demonstrably occur
- **Irreversible actions** — anything that can't be undone if done in the wrong order

Use prompts when:

- **Style and preferences** — formatting, tone, naming conventions
- **Soft guidelines** — "prefer concise responses," "use British spelling"
- **Non-critical order** — steps where doing them out of order causes inconvenience, not harm

The rule of thumb: if violating the rule would require a human to clean up a mess or cause real-world harm, enforce it in code.

## Structured handoff protocols

Multi-step workflows often end with a handoff — either to a human agent, another system, or a downstream process. The agent's context window disappears after the session ends. Whatever the next person or system needs must be packaged explicitly.

A good handoff summary includes everything the recipient needs to take action without having to go back and reconstruct what happened:

```python
handoff_summary = {
    "customer_id": "C-12345",
    "customer_name": "Jane Smith",
    "issue_description": "Customer disputes $150 charge from 2025-03-01",
    "root_cause": "Duplicate billing due to system error during payment retry",
    "attempted_resolution": "Standard refund blocked — amount exceeds $100 auto-approval limit",
    "refund_amount_requested": 150.00,
    "recommended_action": "Approve manual refund of $150 — verified duplicate charge",
    "evidence": "Order #98765 shows two identical charges 3 minutes apart"
}
```

Notice what's included: not just the outcome, but the *reasoning*, the *evidence*, and a clear *recommendation*. The human agent receiving this doesn't need to re-investigate — they have everything needed to make a decision.

## Multi-concern workflows

When a customer has multiple issues, an agent shouldn't address them serially with a single context thread. Instead, it should investigate each issue in parallel (using shared context where appropriate), then synthesize a unified resolution.

This avoids the problem of context from issue #1 bleeding into the analysis of issue #2, and it's more efficient — parallel investigation is faster than sequential.

## Key things to know

- Prompts are probabilistic (~85-95% compliance); hooks are deterministic (100% enforcement)
- Use programmatic gates when financial, safety, or compliance consequences are involved
- `PreToolUse` hooks fire before a tool executes — the right place to implement prerequisite checks
- A `deny` decision blocks the tool and sends Claude a reason, allowing it to take corrective action
- Structured handoff summaries must include everything the recipient needs — context doesn't carry over
- The distinction isn't about trust in Claude — it's about the cost of the rare failure case

## Exam tips

- Questions describe a scenario and ask whether to use prompts or hooks — the answer is hooks whenever the scenario involves financial transactions, safety, compliance, or irreversible actions.
- Watch for the framing "Claude usually follows this rule" — "usually" is the signal that you need programmatic enforcement.
- Handoff protocol questions often test whether you include reasoning/evidence vs. just the outcome — the answer is always include full context.
- The exam may present hook code and ask what happens when a precondition fails — Claude receives the `permissionDecisionReason` and can adapt its approach.
- Don't confuse enforcement order with sequencing — hooks enforce *preconditions*, not step sequences per se.
