## What are escalation patterns?

An AI agent in a customer-facing or production workflow can't handle every situation autonomously. Some situations need human judgment — either because the customer asked for it, because the agent lacks authority to make a decision, or because the policy is silent on this specific case.

**Escalation** is the agent handing off to a human. The key design challenge is getting the *triggers* right: escalate too eagerly and you undermine the agent's usefulness; escalate too reluctantly and you frustrate customers and make poor decisions.

## When to escalate

Four situations that always warrant escalation:

1. **Customer explicitly requests a human** — honour this immediately, no exceptions
2. **Policy gap** — the policy is silent on this situation (not just complex, but genuinely uncovered)
3. **Inability to make meaningful progress** — reasonable attempts have failed
4. **Authority limit** — resolution requires a policy exception the agent isn't authorised to grant

## Handling explicit human requests

The most common mistake: the agent tries to resolve the issue first despite the customer asking for a human.

```python
# WRONG — agent decides whether the issue "warrants" escalation
if customer_requests_human and issue_seems_simple:
    attempt_resolution_first()  # Never do this

# CORRECT — explicit requests are always honoured
if customer_requests_human:
    escalate_to_human(reason="Customer explicitly requested human agent")
```

There is one permitted nuance: if the agent *can* resolve the issue right now, it may offer to do so before transferring — but if the customer reiterates the request, escalate immediately:

```
Customer: "This is ridiculous, I just want to talk to a real person!"

Agent: "I completely understand. I can actually resolve this right now — would
        you like me to process a full refund of $150? If you'd still prefer to
        speak with a human after that, I'll transfer you immediately."

[If customer reiterates: escalate, no further resistance]
```

## Policy gap escalation

A policy gap is when the policy doesn't cover the specific situation — not when it's complex.

```
Customer: "I found your product for $50 less on a competitor site — will you match it?"

Policy covers:     Own-site price matching
Policy is silent:  Competitor price matching

→ ESCALATE — this is a policy gap requiring human judgment
→ NOT OK: Inventing a policy ("we don't match competitor prices")
→ NOT OK: Granting an exception without authority
```

If the agent invents a policy response, it may contradict the actual policy. The correct response is: "That's a great question. Let me connect you with our team who can help with that."

## Disambiguating multiple customer matches

When a customer lookup returns multiple matches, the agent must *ask* — not guess:

```python
if len(customer_matches) > 1:
    response = """I found multiple accounts matching that name. 
    Could you provide one of the following?
    - Email address on the account
    - Last 4 digits of the payment method
    - Account number"""

# WRONG — selecting "most recent login" or "highest value account"
# is a heuristic that could access the wrong customer's data
```

Heuristic selection risks accessing the wrong customer's data. Disambiguation is a user safety issue, not just an accuracy issue.

## Unreliable escalation proxies

Two commonly misused proxies for escalation:
- **Sentiment**: Frustrated customers may have simple issues. Calm customers may have complex ones. Sentiment predicts *emotion*, not *escalation need*.
- **Self-reported confidence**: Agents that report their own confidence are poorly calibrated — high confidence doesn't mean correct.

Use explicit criteria with examples, not sentiment or confidence scores, to trigger escalation.

## Explicit escalation criteria in the system prompt

```
Escalate when:
1. Customer says "I want to speak to a person" or equivalent → escalate immediately
2. Resolution requires a policy exception you're not authorised to grant → escalate
3. Issue has been open for 3+ interactions without resolution → escalate
4. Account flag escalation_required is set → escalate

Do NOT escalate for:
- Standard returns within the policy window
- Billing questions with clear answers
- Order status inquiries
```

Include few-shot examples of edge cases so the agent learns where the line is.

## Key things to know

- Explicit human requests must be honoured immediately — no "let me try first" override
- Policy gaps (policy is silent) must escalate — agents should not invent policy responses
- Multiple customer matches → always ask for disambiguation, never guess by heuristic
- Sentiment and self-reported confidence are unreliable escalation triggers
- Escalation criteria belong in the system prompt as explicit conditions with examples

## Exam tips

- "Customer asked for a human but agent tried to resolve first" → explicit request should escalate immediately
- "Policy doesn't cover this case" → escalate (policy gap), not "resolve based on similar policy"
- "Agent selected account by most recent login" → wrong; disambiguation requires asking the customer
- Sentiment is not a reliable escalation trigger — this is a deliberate exam trap
