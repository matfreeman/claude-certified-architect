## What are the built-in tools?

Claude Code now exposes more built-in tools than just these six — for example `Agent`, `Skill`, `LSP`, `Monitor`, and in some environments `PowerShell`. For **Domain 2 exam prep**, though, the focus is still the classic file-and-shell tools: **Read**, **Write**, **Edit**, **Bash**, **Grep**, and **Glob**.

Knowing *which* tool to use for which task matters. Using `Bash` with `find` to search files, or `Read` to scan every file for a pattern, is significantly slower and more context-heavy than using the dedicated tools. The built-in tools are optimised for their specific tasks and provide better output formatting.

## How the tools work

```
┌─────────┬──────────────────────────────┬────────────────────────────────┐
│ Tool    │ What it does                 │ When to use                    │
├─────────┼──────────────────────────────┼────────────────────────────────┤
│ Glob    │ Finds files by path pattern  │ "Find all test files"          │
│ Grep    │ Searches file contents       │ "Find where X is called"       │
│ Read    │ Returns full file content    │ "Read this specific file"      │
│ Edit    │ Replaces text with new text  │ "Change this function"         │
│ Write   │ Overwrites entire file       │ Fallback when Edit can't match │
│ Bash    │ Runs shell commands          │ Build, test, git, env setup    │
└─────────┴──────────────────────────────┴────────────────────────────────┘
```

## Glob patterns

Glob matches file *paths*, not file contents. Use it to find files by name or extension:

```
**/*.test.tsx        → All test files in any directory
src/api/**/*         → Every file under src/api/
*.md                 → Markdown files at the project root only
src/components/*.tsx → .tsx files directly in src/components/
```

## The Edit fallback pattern

Edit works by finding a unique block of text in the file and replacing it. If the anchor text appears more than once, Edit fails. When that happens, fall back to `Read → Write`:

```
1. Read the full file into context
2. Identify what needs changing
3. Write the entire file back with the change applied
```

This is more expensive (sends the whole file) so Edit is always preferred for targeted changes.

## Incremental exploration strategy

When exploring a codebase, don't read every file at once. Follow the dependency chain:

```
1. Glob for entry points    → src/index.ts, main.py, etc.
2. Read the entry point     → understand top-level imports
3. Follow imports           → Read each imported module
4. Grep for usages          → find where functions are called
5. Read only what you need  → don't pre-load the whole codebase
```

This keeps context usage manageable on large projects.

## Tracing through wrapper modules

One exam pattern is tracing a function through re-exports or wrappers. Searching for a single function name is often not enough because modules may rename or re-export it.

Use this sequence:

```
1. Grep the source module for export statements
2. Read the wrapper module to see aliases or re-exports
3. Grep for each exported name across the codebase
4. Read the highest-signal callers you find
```

This is how you trace usage reliably when a function passes through helper layers before reaching real application code.

## Key things to know

- Use Glob for path-pattern searches, Grep for content searches — never `bash find` or `bash grep` when these tools exist
- Current Claude Code has more built-ins than these six, but the exam focus for this task statement is still Read/Write/Edit/Bash/Grep/Glob
- Edit requires *unique* anchor text — if the same string appears twice, it fails
- Read → Write is the correct fallback when Edit can't find a unique match
- Incremental exploration (follow imports) beats loading the entire codebase upfront
- For wrapper-heavy code, trace exports first, then search each exported name across the codebase
- Bash is the right tool for running builds, tests, git commands, and environment setup

## Exam tips

- A question about "searching for function usages" → Grep (content), not Glob (paths)
- A question about "finding test files" → Glob (path pattern), not Grep
- Edit fails with duplicate anchor text → the correct fix is Read + Write, not re-trying Edit with different text
- The exam may ask which tool is "most efficient" for a given task — always prefer the dedicated tool over a Bash equivalent
