## What is the agentic loop?

When Claude is given a task — "fix the bug in auth.py", "research this topic", "refactor this module" — it doesn't just produce one response and stop. Instead it enters a cycle: think, act, observe, think again. This cycle is the **agentic loop**.

In the loop, Claude receives the current state of the conversation (including any tool results from previous steps), decides whether it needs more information or can answer directly, and either calls a tool or produces a final answer. If it calls a tool, the tool runs, the result is appended to the conversation, and Claude gets another turn. This continues until Claude decides it's done — or until some safety limit kicks in.

The loop is what makes Claude an *agent* rather than just a chatbot. A chatbot answers in one shot. An agent takes multiple actions, observes the results, and adapts. Understanding the loop is foundational to everything else in the SDK.

## How it works

```
┌─────────────────────────────────────────────────────┐
│                    AGENTIC LOOP                     │
│                                                     │
│   ┌──────────┐    tool_use     ┌──────────────┐    │
│   │          │ ─────────────▶  │  Execute     │    │
│   │  Claude  │                 │  Tools       │    │
│   │          │ ◀─────────────  │              │    │
│   └──────────┘  tool_results   └──────────────┘    │
│        │                                            │
│        │ end_turn / max_tokens / refusal            │
│        ▼                                            │
│   ┌──────────┐                                      │
│   │  Result  │  ◀── loop exits here                 │
│   └──────────┘                                      │
└─────────────────────────────────────────────────────┘
```

Each iteration of the loop looks like this:

1. Claude receives the full conversation history (system prompt + all prior messages + all prior tool results)
2. Claude responds — either with one or more tool calls, or with a plain text final answer
3. If Claude made tool calls: the SDK executes each tool, collects results, appends them to the conversation, and sends everything back to Claude
4. If Claude produced a final answer (no tool calls): the loop ends

The key insight is that **Claude doesn't execute tools itself** — it requests them. The SDK (or your own orchestration code) is what actually runs the tools and feeds results back. Claude only sees text.

### Stop reason values

Every response from Claude carries a `stop_reason` that explains why it stopped generating:

| `stop_reason` | Meaning | What happens next |
|---|---|---|
| `"tool_use"` | Claude wants to call one or more tools | Execute tools, append results, continue loop |
| `"end_turn"` | Claude is done — this is the final answer | Exit loop, use the `result` field |
| `"max_tokens"` | Hit the output token limit mid-response | Handle as error, optionally resume |
| `"refusal"` | Claude declined the request | Handle gracefully |

When you use the SDK, you never need to inspect `stop_reason` directly during execution — the SDK handles the `tool_use` / `end_turn` cycle for you. The `stop_reason` concept matters most for understanding *how* the SDK knows when to keep going vs. stop.

## Using the SDK vs the raw API

With the **raw API**, you would manually check `stop_reason`, extract tool call details, run the tools yourself, format results, and POST back to the API. This is tedious and error-prone.

The **Agent SDK** handles all of that internally. You call `query()`, provide tools, and iterate over the stream of messages as the loop runs. You only need to handle the final `ResultMessage` — and optionally inspect `AssistantMessage` objects if you want to observe intermediate turns.

```python
from claude_agent_sdk import query, ResultMessage, AssistantMessage, ClaudeAgentOptions

async for message in query(
    prompt="Find and fix the bug in auth.py",
    options=ClaudeAgentOptions(
        allowed_tools=["Read", "Edit", "Bash", "Glob", "Grep"],
        max_turns=30,
        effort="high",
    ),
):
    if isinstance(message, AssistantMessage):
        # Optional: observe each turn as it happens
        print(f"Turn completed: {len(message.content)} content blocks")
    if isinstance(message, ResultMessage):
        if message.subtype == "success":
            print(message.result)
        elif message.subtype == "error_max_turns":
            print(f"Hit turn limit. Resume session: {message.session_id}")
        print(f"Total cost: ${message.total_cost_usd:.4f}")
```

The `async for` loop yields messages as the agent works. Most of the time you only care about `ResultMessage` — that's the terminal event. But you can watch `AssistantMessage` objects stream by if you want real-time visibility into what Claude is doing.

## Key patterns

**`max_turns` is a safety limit, not the primary stopping condition.** Claude exits the loop when it decides it's done (`end_turn`). The `max_turns` parameter is there to prevent runaway loops if something goes wrong — it's a ceiling, not a target. Setting it too low (like `5`) risks cutting off legitimate multi-step work.

**Claude decides which tools to call.** You provide the allowed tool list; Claude decides at each turn which tool to use, what arguments to pass, and whether it needs more turns. You don't hardcode a sequence like "first call Read, then call Edit." That kind of fixed sequencing is a different pattern (prompt chaining — covered in lesson 1.6).

**Don't parse Claude's text to detect completion.** Looking for phrases like "Task complete" or "Done" in the assistant's text is fragile. The `stop_reason: end_turn` and `ResultMessage.subtype: success` are the authoritative signals.

### Result subtypes

`ResultMessage` carries a `subtype` field explaining how the loop ended:

| Subtype | Meaning |
|---|---|
| `success` | Finished normally — `result` field has the answer |
| `error_max_turns` | Hit the `max_turns` limit before finishing |
| `error_max_budget_usd` | Hit a cost budget ceiling |
| `error_during_execution` | API failure or cancellation |
| `error_max_structured_output_retries` | Structured output schema validation failed repeatedly |

## Key things to know

- The loop runs until `stop_reason: end_turn` — Claude decides it's done, not your code
- Each turn, Claude receives the *entire* conversation history including all prior tool results
- The SDK executes tools and feeds results back automatically — you don't call tools yourself
- `max_turns` is a safety ceiling, not a completion trigger
- `ResultMessage` is the terminal event; `AssistantMessage` events are intermediate turns
- `stop_reason: tool_use` → SDK executes tools, continues; `stop_reason: end_turn` → loop exits
- A `ResultMessage` with `subtype: error_max_turns` means the agent hit the limit without finishing — consider resuming the session or increasing `max_turns`

## Exam tips

- Questions often present code that checks for text like "Task complete" in assistant output as a stopping condition — this is wrong. The correct signal is `ResultMessage.subtype == "success"`.
- Watch for answers that confuse your role vs. the SDK's role. You configure tools and limits; the SDK handles the loop mechanics and tool execution.
- `max_turns=5` as the *only* exit condition is an anti-pattern — it's a safety limit, not a completion mechanism.
- The `stop_reason` you care about in practice comes from `ResultMessage.subtype`, not from inspecting raw API responses.
- Don't confuse `AssistantMessage` (intermediate, one per turn) with `ResultMessage` (terminal, one per session).
